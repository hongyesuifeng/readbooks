### 摘要
尽管编码智能体进展迅速，但其多模态智能体的进展却滞后。一个关键的挑战是缺乏能够将软件开发的复杂性与深度多模态理解需求相结合的评估测试平台。游戏开发提供了这样一个测试平台，因为智能体必须在视觉游戏场景中导航大型、密集的代码库，同时操纵内在的多模态资产，如着色器、精灵和动画。我们提出了 **GameDevBench**，这是第一个用于评估智能体在游戏开发任务上的基准测试。GameDevBench 包含 132 个源自网页和视频教程的任务。这些任务需要大量的多模态理解且非常复杂——与之前的软件开发基准相比，平均解决方案所需的代码行数和文件更改量是前者的三倍以上。智能体在游戏开发方面仍然面临困难，表现最好的智能体仅解决了 54.5% 的任务。我们发现感知到的任务难度与多模态复杂性之间存在很强的相关性，在以游戏玩法为导向的任务上的成功率从 46.9% 下降到 2D 图形任务上的 31.6%。为了提高多模态能力，我们为智能体引入了两种简单的基于图像和视频的反馈机制。尽管这些方法很简单，但它们持续提升了性能，其中最大的变化是 Claude Sonnet 4.5 的性能从 33.3% 提升到了 47.7%。我们公开发布 GameDevBench 以支持对智能体游戏开发的进一步研究。

### 1. 引言
多模态语言模型（LM）智能体的进展落后于单模态智能体 [39, 13, 43, 17]。智能体游戏开发——尽管其具有固有的多模态性、日益增长的公众兴趣以及人工智能与游戏结合的悠久历史 [33, 23, 27, 26, 12, 10, 37]——在很大程度上被研究界所忽视。大多数先前的工作集中在游戏开发中的特定目标，如替代图形引擎的下一帧预测 [30, 21]、替代功能和外观资产创建的程序化内容生成 [29, 24]，或替代非玩家角色和对手的游戏游玩智能体 [33, 26]。对于通用游戏开发（即在游戏引擎中开发游戏）的智能体应用，几乎没有甚至完全没有研究，这很可能是因为直到最近这看起来还是不可想象的。随着 LM 智能体能力的不断提升，很自然地会问：智能体能开发视频游戏吗？
游戏开发结合了许多理想的特征，使其成为现代智能体领域中一个极具挑战性的基准。首先，任务是复杂且上下文丰富的，项目通常包含大量的文件、资产和文件夹，类似于传统的软件开发 [39]。其次，任务本质上是多模态的，需要视觉理解静态元素（如地图或场景布局）和动态元素（如动画或移动），以准确评估项目状态。最后，任务解决方案可以通过代码确定性地验证，这避免了使用经常存在偏见的 LLM-as-a-Judge [42] 等方法的需要 [34, 18]。例如，可以通过检查每一帧的动画状态来验证是否使用了正确的动画。这些特征的结合使游戏开发成为评估复杂、多模态智能体能力的理想环境。
在这项工作中，我们研究了智能体解决现代游戏引擎复杂游戏开发任务的能力。据我们所知，这是评估该能力的首项工作。游戏开发通常涉及在称为游戏编辑器的 GUI（图形用户界面）中创建和编辑工件，如精灵表动画、碰撞形状、游戏逻辑脚本和场景布局。然后，游戏引擎将这些工件处理成可运行的游戏构建。常见的游戏引擎示例包括 Unity、Unreal Engine 和 Godot，每个都提供编辑器和引擎。游戏开发任务具有欺骗性的复杂性。例如，为一个平台游戏创建一个“简单的”意大利水管工任务，需要为各种状态（如空闲、跳跃或奔跑）创建动画，设置碰撞体以允许跳在乌龟等敌人身上，编写脚本以允许控制，添加动作的音效等。
我们专注于 Godot 环境进行这项工作，原因如下。首先，Godot 在 MIT 许可证下完全开源，这使得它易于扩展并与基准测试一起发布。其次，Godot 是一个日益流行的游戏开发引擎，2024 年和 2025 年在 Steam 上分别有 770 和 1185 个发布版本。第三，Godot 的环境与 Unity 非常相似，而 Unity 是目前最流行的游戏开发引擎。最后，Godot 项目（不包括图像等资产）可以用代码表示，这使得扩展现有的 LLM 智能体能力变得简单，而无需构建特定的工具使用 API。
我们提出了 GameDevBench，这是第一个用于评估智能体解决游戏开发任务能力的基准测试。任务是通过分析和处理 Godot YouTube 和网页教程创建的。这些教程涵盖了广泛的主题，如 2D 精灵动画、角色控制器（即角色移动）、碰撞器和平台、着色器使用、粒子效果等。这确保了任务不仅多样化，而且符合常见的游戏开发需求。任务非常复杂且内容丰富。它们不仅需要深入理解各种文件类型和资产（例如图像），而且平均而言，任务所需的代码行数更改量比 SWE-Bench [39] 多三倍以上。对于每个任务，智能体会获得一个包含代码和各种资产的项目文件夹，以及软件基准测试中标准的指令 [39]。任务成功与否通过 Godot 脚本框架内构建的测试进行评估。这使我们能够确定性地测试物理或多边形形状等特征。此外，每个任务都附带一个经过验证的参考解决方案。GameDevBench 的所有代码和任务项目文件均已公开发布。
我们发现，虽然智能体能力日益增强，但它们仍然难以应对大多数游戏开发任务。在没有额外支持的情况下，表现最好的智能体仅成功完成了 47.0% 的任务。特别是，当任务需要增加多模态理解时，模型的表现明显更差。例如，智能体在处理以游戏玩法为导向的任务时的表现几乎是在图形任务上的两倍。
为了提高智能体的多模态能力，我们提出了两种方法，在解决任务时为智能体提供多模态反馈。一种方法通过模型上下文协议（MCP）服务器 [1] 提供编辑器当前状态的截图视图，另一种方法录制游戏场景的视频。尽管它们很简单，我们发现这两种方法在经验上都是有效的，几乎提高了所有模型的智能体性能。

### 2. 基准测试构建
GameDevBench 包含从在线教程中提炼出的游戏开发任务（例如，“使用给定的精灵表添加行走动画”）。GitHub 仓库是丰富的数据来源，但可能嘈杂且文档不全。此外，与之前关于基准测试通用软件开发的工作 [13] 不同，没有明显的流行开源游戏仓库可供选择。然而，游戏开发社区创建了大量的在线教程——其中许多附带解决方案仓库——指导开发者完成常见的开发用例。我们使用多步骤流程，利用这些教程构建游戏开发任务。
#### 2.1 阶段 1：数据准备
游戏开发教程主要以文本或视频格式提供。对于所有教程，我们搜索并筛选，仅包含带有宽松开源许可证的相应 GitHub 仓库的 Godot 4 教程。
*   **视频。** 我们从 YouTube 获取视频教程。为了将视频转换为文本，我们使用流行的 YouTube 转录 API 提取每个视频的文本转录。为了搜索匹配的 GitHub 仓库，我们解析视频描述中的任何 GitHub 链接。最终结果是为每个教程创建一个文件夹，包含转录和相应的 GitHub 仓库。我们根据观看次数选择了 102 个视频教程进行处理。每个教程平均包含 29 分钟的内容。最终，我们使用了 57 个教程，因为并非所有教程都可用，原因包括仓库不可用或 Godot 版本标记错误。
*   **网页。** 对于网页文本教程，我们取材于“Godot Recipes by KidsCanCode” [16]，该资源列在 Godot 官方文档的社区资源页面上。我们使用 Python 脚本抓取网页，目的是镜像处理后视频教程文件夹的结构。最终结果是 99 个教程文件夹，每个包含教程文本内容、从网页下载的视觉数据媒体目录、GitHub 仓库以及包含教程 URL 等信息的元数据 JSON。最后，我们要求 LLM 根据任务创建的适用性对教程进行排序，并使用排名前 31 的教程进行后续任务构建。
#### 2.2 阶段 2：自动任务构建
给定教程文件夹，要求智能体创建任务，其中：1) 指令遵循教程内容，2) 任务文件直接基于仓库中的现有文件创建，3) 单元测试必须仅测试指令中明确要求的特征。访问解决方案仓库至关重要，因为它允许智能体创建它通常没有能力解决或创建的任务。根据智能体的判断，每个教程被拆分为多个任务以捕捉更明确的技能。例如，智能体可以将平台游戏教程分解为角色动画、控制和碰撞器以及瓦片地图构建等任务。我们使用 GPT-5 系列模型的 Codex 智能体从每个教程构建任务。选择 Codex 主要是因为当时的 API 限制和可用性；我们没有注意到诸如 Claude Code 等智能体之间的显著差异。我们创建了 202 个初始任务，平均每个教程 1.3 个任务。完整的提示词可在附录 A 中找到。
#### 2.3 阶段 3：任务细化
在阶段 2 之后，我们发现大多数任务在高层上是合理的（即任务指令是合理的且与教程匹配）。然而，与之前的工作 [7] 类似，智能体无法完美地创建任务和测试。我们对 41 个任务的一个小子集进行了初步研究，人工标注员审查任务并记录观察到的问题。研究发现，43% 的任务没有问题，50% 的任务存在需要轻微更新的问题，如场景在摄像机外、测试断言不存在的指令或意外引用教程的其他部分，7% 的任务包含重大问题难以修复。由于大多数错误都是轻微且容易发现的，我们采用了混合流程来细化任务。基于初步研究，我们构建了提示词和检查表来捕捉最常见的错误。然后，我们雇佣了一个智能体根据检查表自动验证并修复这些错误。我们在处理所有未来的教程和任务时重用了这个提示词（附录 B）。
#### 2.4 阶段 4：人工标注
最后，8 名人工标注员（其中 5 名具有游戏开发经验）审查了所有任务。标注有两个主要目标。首先是确保任务经过正确性和可解决性的验证，这是常见做法 [39, 7]。指示标注员查找并修复任何模棱两可的指令、冲突的指令和过于严格的测试。此外，要求标注员标记并删除有任何其他问题的任务。其次，我们要求标注员创建现有任务的变体，类似于之前的工作 [43]。一个例子是两个任务在请求的精灵表中使用的动画上有所不同（例如，选择行走还是奔跑动画帧）。总共，我们创建了 115 个基础任务和 17 个任务变体。标注说明可在附录 C 中找到。

### 3. GameDevBench
游戏开发位于创意表达和软件开发的交叉点。因此，GameDevBench 具有多样化的任务集，这些任务本质上是多模态的、复杂且上下文丰富的。
#### 3.1 任务类别
据我们所知，目前还没有在游戏编辑器或游戏引擎中执行的游戏开发任务分类法。为了更好地理解我们的任务多样性并进行更深入的分析，我们沿两个轴对每个任务进行分类。
*   **按技能集分类。** 我们根据每个任务所需的基础游戏开发技能归纳任务分类（表 5）。具体来说，我们采用自下而上的分类程序：首先通过要求 GPT-5-mini 标记每个任务来获取细粒度的技能标签。然后通过单独的请求将标签抽象为更高层次的技能类别。随后，游戏开发者审查并细化这些类别以确保一致性和有效性。这个过程产生了四个技能类别：**2D 图形与动画**、**3D 图形与动画**、**游戏逻辑**和**用户界面**。
*   **按编辑器类型分类。** Godot 包含几种不同类型的编辑器，用户使用它们来解决各种任务。Godot 中主要有三种类型的编辑器。场景编辑器（图 3，左上）允许用户通过构建关卡地图或放置和编辑对象来修改游戏场景。脚本编辑器（图 3，右上）是一个内置的代码编辑器。上下文编辑器根据用户正在编辑的内容出现在底部面板（图 3，底部）。例如，当编辑动画资源时，动画编辑器将出现。一些最常见的上下文编辑器包括动画、音频、着色器和瓦片集编辑器。我们通过询问 GPT-5-mini 确定用户解决任务所需的编辑器，对基准测试中的每个任务进行分类。虽然智能体可能不直接与这些编辑器交互，但用户使用的编辑器类型为任务分类提供了强有力的代理。
为了简单起见，我们为每个任务分配一个技能类别和一个编辑器类型。我们在附录 E 中提供了多个示例及其技能类别和编辑器类型。
#### 3.2 GameDevBench 的特征
GameDevBench 有一组独特的特征，描述如下。我们在附录 F 中提供了额外的任务统计数据。
*   **跨任务的多样化文件类型。** 与软件领域的智能体基准测试 [13] 不同，GameDevBench 要求智能体处理跨各种模态的各种文件类型（图 4，左）。事实上，绝大多数任务（82.4%）包含额外的资产，如图像、文本字体、着色器、音频 和其他资源。因此，GameDevBench 本质上测试了智能体的多模态能力。
*   **多样化的任务类型。** 虽然还有其他领域（如前端开发 [44, 25] 或幻灯片生成 [6]）与多模态和代码生成交叉，但大多数这些领域专注于类似于用户界面生成的任务。另一方面，GameDevBench 具有多样化的任务集。在 132 个基准任务中，领域分布如下：(35.6%) 游戏逻辑，(25.7%) 3D 图形与动画，(19.7%) 2D 图形与动画，以及 (15.9%) 用户界面。
*   **复杂且上下文丰富的解决方案。** 与软件任务类似，GameDevBench 解决方案需要跨位置的编辑，将多个文件编织在一起。我们的参考解决方案平均涉及 5 个文件的更改和 106.2 行代码，跨越 3.4 种不同的文件类型（图 4，右）。与 SWE-Bench [13] 相比，这所需的代码行数和文件更改量是三倍以上，表明任务和相应解决方案具有实质性的复杂性。
*   **多模态解决方案的确定性验证。** 评估多模态解决方案本质上是具有挑战性的，解决方案通常通过 CLIP [22] 或视觉 LLM-as-a-Judge [41] 等指标进行评估。然而，这些方法要么是正确性的代理，要么是非确定性的。GameDevBench 改用 Godot 的测试框架，允许我们直接测试游戏行为。例如，我们可以通过单元测试检查对象是否在摄像机视野内，或对象碰撞器是否已交互。因此，测试是可重复且可验证的，类似于软件基准测试，同时测试多模态问题。
*   **灵活的解决方案方法。** 虽然测试是确定性的，但得出解决方案的方法是灵活的。在这项工作中，为了简单起见，我们评估仅通过代码生成尝试解决任务的智能体。然而，在编辑器中直接解决每个任务同样可行，方法更类似于计算机使用。我们基于测试的验证允许直接比较不同的解决策略。
*   **可持续更新。** 虽然这不是我们基准测试独有的，但我们的流程是可重复的，因此基准测试可以持续更新。人工验证工作量很小，每个任务的验证时间不到 10 分钟。

### 4. 评估
我们在 GameDevBench 上评估了各种模型和智能体框架。
*   **模型选择。** 从 Claude 模型家族中，我们评估了 Claude Haiku 4.5、Claude Sonnet 4.5 和 Claude Opus 4.5。从 Gemini 家族中，我们评估了 Gemini 3 Flash 和 Gemini 3 Pro。我们从 ChatGPT 家族评估了 ChatGPT Codex 5.1。对于开源权重模型，我们评估了 Qwen3-Vl-235B-Instruct 和 Kimi K2.5。
*   **智能体框架选择。** 为了允许智能体访问项目文件和 Godot 应用程序本身，我们专注于在本地运行的智能体框架。我们选择命令行界面（CLI）智能体框架，因为它们能够直接读取代码、图像和其他资产文件。我们在各自的智能体框架中评估每个模型——Claude 模型使用 claude-code，Gemini 模型使用 gemini-cli，ChatGPT 模型使用 codex。我们使用 OpenHands [35] 评估 Kimi K2.5 和 Qwen3-Vl。我们还使用 OpenHands 评估了 Claude Sonnet 4.5、Gemini 3 Flash 和 ChatGPT Codex 5.1，以比较跨框架的性能。
#### 4.1 多模态反馈
在此，我们概述了两种工具配置，允许智能体通过编辑器截图和/或渲染视频从 Godot 访问更丰富的多模态信息。
*   **基线。** 作为基线，每个智能体从项目目录内开始，并获得任务指令以及如何运行 Godot 的基本说明。我们提供额外的方法来支持智能体，主要是为了观察额外的视觉上下文是否提高性能。我们在附录 D 中提供了完整的提示词。
*   **编辑器截图 MCP。** 我们开发了一个 MCP 服务器，它加载当前任务的 Godot 编辑器，对编辑器进行截图，然后将图像返回给智能体。这允许智能体查看游戏场景、节点树、节点检查器以及编辑器中存在的其他信息。这种方法允许智能体利用额外的视觉反馈来验证其解决方案。
*   **运行时视频。** 我们为智能体提供有关如何使用 Godot 内置录制功能生成游戏视频的说明，否则该功能经常被忽略或误用。这与 MCP 服务器不同，因为它捕获了 a) 仅在视频中存在的时序元素，以及 b) 当前摄像机视图（编辑器不显示摄像机视图）。通常，模型使用 python 将视频处理为图像帧，而不是直接摄取视频。
#### 4.2 结果讨论
我们现在讨论在 GameDevBench 上评估智能体的发现（表 2）。
*   **即使对于最有能力的模型，游戏开发也具有挑战性，且随着远离前沿模型，性能迅速下降。** 三个不同商业模型家族（GPT、Claude 和 Gemini）中最大的模型在其原生智能体框架中没有额外多模态反馈的情况下，基线性能分别为 34.1%、39.4% 和 46.2%。随着我们远离前沿模型，性能显著下降。Claude Haiku 4.5 解决了 22.0%，Kimi K2.5 解决了 34.1%，而 Qwen3-Vl-235B-Instruct 几乎完全无能为力，仅解决了 8.3% 的任务。相比之下，Qwen3-Vl-235B-Instruct 在前端基准测试 Design2Code [25] 中解决了 92% 的任务。
*   **智能体性能在技能和编辑器类别上差异显著。** 我们观察到一个普遍趋势，即智能体在多模态要求更高的任务上表现更差（图 5）。在技能方面，智能体在游戏逻辑任务上表现最好（平均成功率 46.9%），而在 2D 图形与动画上表现最差（平均成功率 31.62%），后者需要智能体理解图像或其他资产以进行动画和效果。跨编辑器类别的性能则取决于模型能力。整体表现最好的四个模型——Gemini 3 Pro、Gemini 3 Flash、Claude Opus 4.5 和 Claude Sonnet 4.5——在任务上表现相似，无论所需的编辑器类型如何。然而，Claude Haiku 4.5、ChatGPT Codex 5.1 和 Kimi K2.5 在需要场景和上下文编辑器的任务上表现较差，这些任务通常比脚本任务对多模态要求更高。
*   **多模态工具持续提升智能体性能。** 我们发现为智能体提供 MCP 或视频指令可以提高性能。这一趋势几乎适用于所有模型（除了 Claude Haiku 4.5，其性能保持相似）。然而，表现最好的方法因模型而异。例如，Gemini 3 Flash 从 MCP 中受益，从 47.0% 提高到 50.0%，而在使用视频时为 48.5%。另一方面，Claude Sonnet 4.5 使用视频从 33.3% 提高到 41.7%，而在使用 MCP 时没有看到改进。同时使用 MCP 和视频提供的额外益处可以忽略不计，达到与两者中最佳方法相似的性能。虽然所有任务都可以通过代码验证，但视觉反馈允许智能体验证和修正错误。这种行为与最近的工作 [41] 中看到的非常相似，其中视觉反馈提高了智能体性能。
*   **智能体框架的选择会对性能产生重大影响，但效果取决于模型。** 我们使用原始框架和 OpenHands 评估了 Claude Sonnet 4.5、Gemini 3 Flash 和 GPT 5.1 Codex（表 2）。我们观察到，当从原生框架切换到 OpenHands 时，Claude Sonnet 4.5 和 GPT 5.1 Codex 的性能分别从 33.3% 提高到 43.2% 和从 34.1% 提高到 45.5%。另一方面，Gemini 3 Flash 的性能从 47.0% 下降到 36.4%，使其在原生智能体框架中成为三个模型中最好的，而在 OpenHands 中成为最差的。这可能是由于 Gemini 模型与 OpenHands 之间的编辑工具不兼容。
*   **成本因模型、框架以及是否提供多模态反馈而异。** 我们发现启用多模态反馈几乎总是会增加成本，以换取提高的性能（图 6）。唯一的例外是 Qwen3-Vl，它在基准测试上表现挣扎。令人惊讶的是，同时使用两种方法在所有 claude-code 智能体中更具成本效益，同时仍保持提高的性能。我们怀疑模型能够在执行过程中动态选择更有效的反馈机制。我们发现 Gemini 3 Flash 是最具成本效益的模型，而 Claude 模型往往是最不具成本效益的。我们观察到模型容量和每个 token 成本并不一定与最终任务成本相关。例如，在使用 claude-code 时，Claude Opus 4.5（图 6，黄色）的成本是 Claude Sonnet 4.5（图 6，绿色）的一半，尽管提供了改进的性能。另一个例子是，在使用 OpenHands 时，Claude Haiku 4.5 的成本与 ChatGPT Codex 5.1 大致相同，尽管每个 token 更便宜且表现显著更差。
#### 4.3 错误分析与改进方向
我们手动分析了智能体在解决任务时犯的一些最常见错误。这些错误表明了能力的潜在差距和智能体开发的未来方向。虽然有多种错误，但我们观察到两种一致的错误模式。
*   **智能体在多模态理解上表现挣扎。** 也许最一致的错误模式源于缺乏多模态理解。具体来说，通常需要理解多模态输入才能正确完成游戏开发任务。例如，创建（甚至仅仅是选择）一个动画需要智能体解析多个图像或在精灵表中挑选特定的精灵。目前，智能体经常挑选错误的图像或精灵（例如，选择了行走动作的精灵而不是攻击动作的精灵）。显然，多模态理解的改进将显著提高智能体游戏开发的性能。
*   **智能体在常见的游戏开发模式上表现挣扎。** 在游戏开发中，有许多常见的开发模式。例如，游戏元素（在 Godot 中称为节点）形成树结构，其中特定节点如 AnimatedSprite2D 和 CapsuleCollider 分别处理动画和物理。另一个例子是信号，当满足条件（如两个碰撞器相交）时，在各个文件之间触发。智能体经常将节点添加到树中的错误层级，丢弃必要的信号，或将资源分配给错误的元素。我们在附录 G 中提供了此类错误的示例。这加强了模型训练中长期存在的趋势——模型必须在特定领域上进行训练才能在该领域表现出色。

### 5. 相关工作
**智能体基准测试。** 软件开发一直是智能体开发的首要前沿之一。SWE-Bench [13, 38] 可能是第一个基准测试，也是智能体软件开发的催化剂。随着时间的推移，已经开发了多个新的软件基准测试 [5, 19, 40]，但它们在很大程度上是单模态的。少数多模态软件基准测试主要集中在 JavaScript 前端开发 [44, 25, 39]。相反，多模态智能体最常见的用例一直是计算机使用 [36] 和网络导航 [43, 17]。该领域的进展具有挑战性，因为智能体必须在动作空间中操作，而不仅仅是编写代码。游戏开发通过需要多模态输入来弥合这些领域之间的差距，但允许代码输出。GameDevBench 能够有效地利用软件和计算机使用领域的优势，从而实现有效的多模态评估。
**游戏玩法。** 人们一直对人工智能（AI）在游戏中的应用有着浓厚的兴趣 [11]；游戏玩法被视为 AI 系统能力或智能的代理，项目范围从 Deep Blue [4]、Alpha Go [26] 和 Cicero [9] 到最近的 SIMA 2 [2] 等通用智能体。实际上，游戏提供了一个具有明确奖励信号的交互式模拟环境，允许研究人员试验方法——特别是强化学习——以提高模型能力。最近涌入的 LLM 玩 Pokémon [14, 15, 8] 使用游戏游玩智能体来评估和探索前沿模型的智能体推理能力，这随后直接用于游戏开发以测试游戏 [20]。这种从游戏游玩智能体作为游戏中的 NPC 和对手转变为游戏开发过程的一部分的过渡，标志着对诸如 GameDevBench 等基准测试的及时需求。
**游戏开发。** Concordia [31] 及随后关于桌面角色扮演游戏的工作 [32] 试图用完全通过与 LLM 交互创建的高度自适应故事来替代可交互角色。其他工作试图完全替换游戏的物理引擎，根据玩家动作立即生成帧 [3]。程序化内容生成有着利用 AI 进行游戏资产创建 [29, 24] 和进化关卡设计 [28] 的悠久历史。然而，这些主要集中在游戏开发的单一方面。最终，每个功能仍然需要在游戏引擎中组合以开发完整的游戏，这正是 GameDevBench 直接评估的能力。
---
### 6. 结论
我们提出了 GameDevBench，这是第一个用于评估智能体解决游戏开发任务能力的基准测试。我们发现智能体在游戏开发任务中表现挣扎，尤其是当任务需要更深层次的多模态理解时。前沿模型与非前沿模型之间的差距巨大，pass@1 的绝对差异高达 46.2%。此外，我们观察到智能体框架可以极大地影响模型性能，尽管这高度依赖于模型和框架。最后，我们表明即使是简单的工具来提供多模态反馈也能持续提高智能体性能，提供了高达 42% 的相对提升。我们的发现强调了提高智能体多模态能力的必要性——无论是通过训练还是视觉反馈方法。我们推测，解决这些需求将提高智能体甚至在软件和游戏开发之外领域的性能。
