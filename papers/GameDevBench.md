标题：GameDevBench：通过游戏开发评估智能体能力
作者：Wayne Chi¹, Yixiong Fang¹, Arnav Yayavaram¹, Siddharth Yayavaram¹, Seth Karten², Qiuhong Anna Wei¹, Runkun Chen¹, Alexander Wang¹, Valerie Chen¹, Ameet Talwalkar¹, 和 Chris Donahue¹

摘要
尽管编程智能体进展迅速，但其多模态对应物的进展却相对滞后。一个关键挑战是缺乏将软件开发的复杂性与深度多模态理解需求相结合的评估测试平台。游戏开发提供了这样一个测试平台，因为智能体必须在大型、密集的代码库中导航，同时在视觉游戏场景中操作着色器、精灵和动画等内在多模态资产。我们提出了 GameDevBench，这是第一个用于评估智能体在游戏开发任务上表现的基准。GameDevBench 包含132个源自网络和视频教程的任务。这些任务需要显著的多模态理解，并且非常复杂——平均解决方案所需的代码行数和文件更改量是先前软件开发基准的三倍以上。智能体在游戏开发上仍然挣扎，表现最好的智能体仅解决了54.5%的任务。我们发现，感知到的任务难度与多模态复杂性之间存在强相关性，成功率从游戏玩法导向任务的46.9%下降到2D图形任务的31.6%。为了提高多模态能力，我们为智能体引入了两种简单的基于图像和视频的反馈机制。尽管简单，这些方法持续提高了性能，最大的变化是 Claude Sonnet 4.5 的性能从33.3%提升到了47.7%。我们公开发布 GameDevBench，以支持对智能体游戏开发的进一步研究。

1. 引言
多模态语言模型（LM）智能体的进展滞后于其单模态对应物。智能体游戏开发——尽管具有内在的多模态性、日益增长的公众兴趣以及人工智能与游戏相结合的丰富历史——但在很大程度上被研究社区所忽视。大多数先前的工作集中在游戏开发中的特定目标，例如下一帧预测（取代图形引擎）、程序化内容生成（取代功能和外观资产创建）或游戏玩家智能体（取代非玩家角色和对手）。几乎没有关于智能体用于一般游戏开发（即在游戏引擎内开发游戏）的研究，这可能是因为直到最近这似乎还是不可想象的。随着 LM 智能体能力的持续提升，自然而然地会问：智能体能开发视频游戏吗？
游戏开发结合了许多理想的特性，使其成为现代智能体领域中一个极具挑战性的基准。首先，任务复杂且上下文丰富，项目通常跨越大量文件、资产和文件夹，类似于传统软件开发。其次，任务内在多模态，需要对静态元素（例如地图或场景布局）和时间动态（例如动画或移动）进行视觉理解，以准确评估项目状态。最后，任务解决方案可以通过代码进行确定性验证，这缓解了使用“LLM作为评判者”等方法的需求，这些方法通常存在偏见。例如，可以通过检查每一帧的动画状态来验证是否使用了正确的动画。这种特性组合使游戏开发成为评估复杂、多模态智能体能力的理想环境。
在这项工作中，我们研究了智能体为现代游戏引擎解决复杂游戏开发任务的能力。据我们所知，这是第一项评估此能力的工作。游戏开发通常涉及在称为游戏编辑器的图形用户界面（GUI）中创建和编辑精灵表动画、碰撞形状、游戏逻辑脚本和场景布局等工件。然后，游戏引擎将这些工件处理成可运行的游戏构建。常见的游戏引擎示例包括 Unity、Unreal Engine 和 Godot，每个引擎都提供了编辑器和引擎。
游戏开发任务具有欺骗性的复杂性。例如，为平台游戏创建一个“意大利水管工”的“简单”任务将需要为各种状态（如空闲、跳跃或跑步）创建动画，设置一个对撞机以允许在乌龟等敌人身上跳跃，编写脚本以允许控制，为动作添加音效等。我们专注于 Godot 环境有几个原因。首先，Godot 在 MIT 许可证下完全开源，这使得它易于扩展和与基准一起发布。其次，Godot 是一个日益流行的游戏开发引擎，2024年和2025年在 Steam 上分别有770和1185个发布版本。第三，Godot 的环境与 Unity 非常相似，而 Unity 是迄今为止最流行的游戏开发引擎。最后，Godot 项目（不包括图像等资产）可以用代码表示，这使得扩展现有的 LM 智能体能力变得简单，而无需构建特定的工具使用 API。
我们提出了 GameDevBench，这是第一个评估智能体解决游戏开发任务能力的基准。任务是通过分析和处理 Godot YouTube 和网络教程创建的。这些教程涵盖了广泛的主题，如2D精灵动画、角色控制器（即角色移动）、碰撞器和平台、着色器使用、粒子效果等。这确保了任务不仅多样化，而且符合常见的游戏开发需求。任务极其复杂且内容丰富。它们不仅需要深刻理解各种文件类型和资产（例如图像），而且任务平均需要的代码更改行数是 SWE-Bench 的三倍以上。对于每个任务，智能体会获得一个包含代码和各种资产的项目文件夹，以及一条指令，这是软件基准中的标准做法。任务成功与否使用在 Godot 脚本框架内构建的测试来评估。这允许我们确定性测试物理或多边形形状等功能。此外，每个任务都附带一个经过验证的参考解决方案。GameDevBench 的所有代码和任务项目文件都公开发布。
我们发现，虽然智能体的能力越来越强，但它们仍然在大多数游戏开发任务上挣扎。在没有额外支持的情况下，最好的智能体仅成功完成了47.0%的任务。特别是，当任务需要增加多模态理解时，模型的表现显著更差。例如，智能体在处理游戏玩法导向任务时的表现几乎是处理图形任务时的两倍。为了提高智能体的多模态能力，我们提出了两种在解决任务时为智能体提供多模态反馈的方法。一种方法通过模型上下文协议（MCP）服务器提供编辑器当前状态的屏幕截图视图，而另一种方法记录游戏场景的视频。尽管简单，但我们发现这两种方法在实证上都有效，几乎在所有模型上都提高了智能体的性能。
2. 基准构建
GameDevBench 由从在线教程中提炼的游戏开发任务组成（例如，“使用给定的精灵表添加行走动画”）。GitHub 仓库是丰富的数据源，但可能嘈杂且文档不良。此外，与先前关于通用软件开发基准的工作不同，没有明显流行的开源游戏仓库可供选择。然而，游戏开发社区创建了大量在线教程——其中许多带有解决方案仓库——指导开发者完成常见的开发用例。我们使用多步骤流水线利用这些教程构建游戏开发任务。
2.1 阶段1：数据准备
游戏开发教程主要有文本或视频格式。对于所有教程，我们搜索并筛选仅包含具有相应 GitHub 仓库、采用宽松开源许可证的 Godot 4 教程。
视频：我们从 YouTube 获取视频教程。为了将视频转换为文本，我们使用流行的 YouTube 转录 API 从每个视频中提取文本转录。为了搜索匹配的 GitHub 仓库，我们解析视频描述中的任何 GitHub 链接。最终结果是每个教程对应一个文件夹，包含转录和相应的 GitHub 仓库。我们根据观看量选择了102个视频教程。每个教程平均有29分钟的内容。最终，我们使用了57个教程，因为并非所有教程都可用，原因是仓库不可用或 Godot 版本标注错误。
网络：对于网络文本教程，我们从“Godot Recipes by KidsCanCode”获取，该网站列在 Godot 官方文档的社区资源页面上。我们使用 Python 脚本抓取网页，目标是镜像已处理视频教程文件夹的结构。最终得到99个教程文件夹，每个包含教程文本内容、从网页下载的视觉数据媒体目录、GitHub 仓库以及包含教程 URL 等信息的元数据 JSON。最后，我们要求 LLM 根据任务创建的适用性对教程进行排序，并使用前31个教程进行后续任务构建。
2.2 阶段2：自动任务构建
给定教程文件夹，要求智能体创建任务，其中：1）指令遵循教程；2）任务文件直接基于仓库中的现有文件创建；3）单元测试必须仅测试指令中明确要求的功能。访问解决方案仓库至关重要，因为它允许智能体创建其通常没有能力解决或创建的任务。根据智能体的判断，每个教程被拆分为多个任务以捕获更明确的技能。例如，智能体可以将平台游戏教程拆分为角色动画、控制和碰撞器以及瓦片地图构建等任务。我们使用 Codex Agent 和 GPT-5 系列模型从每个教程构建任务。选择 Codex 主要是因为当时其 API 限制和可用性；我们没有注意到 Claude Code 等智能体之间有任何显著差异。我们创建了202个初始任务，平均每个教程1.3个任务。完整的提示可以在附录 A 中找到。
2.3 阶段3：任务精炼
在阶段2之后，我们发现大多数任务在高层次上是合理的（即任务指令是合理的，并与教程匹配）。然而，与先前的工作类似，智能体无法完美创建任务和测试。我们在一小部分41个任务上进行了初步研究，人工标注员审查任务并记录观察到的任何问题。研究发现43%的任务没有问题，50%的任务有需要次要更新的问题，如场景在镜头外、测试断言不存在的指令或意外引用教程的其他部分，7%的任务包含使其难以修复的主要问题。由于大多数错误是次要的且易于捕捉，我们采用混合流程来精炼任务。根据初步研究，我们构建了提示和检查表以捕获最常见的错误。然后，我们雇佣了一个智能体根据检查表自动验证和修复这些错误。我们在处理所有未来的教程和任务时重用此提示（附录 B）。
2.4 阶段4：人工标注
最后，8名人工标注员（其中5人有游戏开发经验）审查了所有任务。标注有两个主要目标。首先是确保任务经过正确性和可解决性验证，这是常见做法。标注员被指示查找并修复任何模棱两可的指令、相互冲突的指令和过于严格的测试。此外，要求标注员标记并删除任何有其他问题的任务。其次，我们要求标注员创建现有任务的变体，类似于先前的工作。一个例子是两个任务，它们根据精灵表中请求的动画不同而有所区别（例如，选择行走与跑步动画帧）。总共，我们创建了115个基础任务和17个任务变体。标注说明可以在附录 C 中找到。
3. GameDevBench
游戏开发位于创意表达和软件开发的交汇点。因此，GameDevBench 具有一组多样化的任务，这些任务内在多模态、复杂且上下文丰富。
3.1 任务类别
据我们所知，目前没有在游戏编辑器或游戏引擎中执行的游戏开发任务的现有分类法。为了更好地理解我们的任务多样性并启用更深入的分析，我们沿着两个轴对每个任务进行分类。
按技能集分类：我们根据每个任务所需的底层游戏开发技能归纳任务分类（表5）。具体而言，我们采用自底向上的分类过程：首先通过要求 GPT-5-mini 标记每个任务来获得细粒度的技能标注。然后通过向 GPT-5-mini 的单独请求将标签抽象为更高层的技能类别。这些类别随后由游戏开发者审查和精炼，以确保一致性和有效性。此过程产生四个技能类别：
游戏玩法逻辑：专注于在 Godot 中实现游戏规则和行为（如运动和碰撞）的任务。示例：敌人AI状态、信号驱动事件、碰撞检测器、角色控制器。占比：35.6%。
3D图形与动画：涉及在 Godot 中构建、渲染和动画化三维场景的任务。示例：材质调整、骨骼动画、相机装置。占比：25.7%。
2D图形与动画：涉及在 Godot 中创建、渲染和动画化二维视觉内容的任务。示例：精灵动画、TileMap设置、2D着色器效果。占比：19.7%。
用户界面：涉及在 Godot 中设计和实现交互式用户界面的任务。示例：HUD布局、菜单导航、UI主题。占比：15.9%。
按编辑器类型分类：Godot 包含多种类型的编辑器，用户使用它们来解决不同的任务。Godot 内主要有三种编辑器类型：
场景编辑器（图3，左上）：允许用户通过构建关卡地图或放置和编辑对象来修改游戏场景。
脚本编辑器（图3，右上）：内置代码编辑器。
上下文编辑器：根据用户正在编辑的内容出现在底部面板（图3，底部）。例如，在编辑动画资源时，动画编辑器会出现。一些最常见的上下文编辑器包括动画、音频、着色器和瓦片集编辑器。
我们通过要求 GPT-5-mini 确定用户解决任务所需使用的编辑器来对基准中的每个任务进行分类。虽然智能体可能不会直接与这些编辑器交互，但用户将使用的编辑器类型为任务分类提供了强有力的代理。为简单起见，我们为每个任务分配一个技能类别和一个编辑器类型。我们在附录 E 中提供了多个示例及其技能类别和编辑器类型。
3.2 GameDevBench 的特点
GameDevBench 有一组独特的功能，我们描述如下。我们在附录 F 中提供了额外的任务统计信息。
跨任务的多样化文件类型：与软件领域的智能体基准不同，GameDevBench 要求智能体处理跨多种模态的各种文件类型（图4，左）。事实上，绝大多数任务（82.4%）包含额外的资产，如图像、文本字体、着色器、音频和其他资源文件。因此，GameDevBench 内在地测试了智能体的多模态能力。
多样化的任务类型：虽然存在其他与多模态和代码生成交叉的领域（如前端开发或幻灯片生成），但大多数这些领域专注于类似于用户界面生成的任务。另一方面，GameDevBench 具有多样化的任务集。在132个基准任务中，领域分布如下：游戏玩法逻辑（35.6%）、3D图形与动画（25.7%）、2D图形与动画（19.7%）和用户界面（15.9%）。
复杂且上下文丰富的解决方案：与软件任务类似，GameDevBench 解决方案需要将多个文件编织在一起的多位置编辑。我们的参考解决方案平均编辑5个文件和106.2行代码，涉及3.4种不同的文件类型（图4，右）。这比 SWE-Bench 所需的代码行数和文件更改量多出三倍以上，表明任务和相应解决方案具有实质性复杂性。
多模态解决方案的确定性验证：评估多模态解决方案本质上是具有挑战性的，解决方案通常通过 CLIP 或视觉 LLM 作为评判者等指标进行评估。然而，这些方法要么是正确性的代理，要么是非确定性的。GameDevBench 改用 Godot 的测试框架，这允许我们直接测试游戏行为。例如，我们可以检查对象是否在相机视野内，或者对象碰撞器是否仅通过单元测试就进行了交互。因此，测试是可重复和可验证的，类似于软件基准，同时测试多模态问题。
灵活的解决方法：虽然测试是确定性的，但得出解决方案的方法是灵活的。在这项工作中，为简单起见，我们评估仅尝试通过代码生成来解决任务的智能体。然而，直接在编辑器中使用更类似于计算机使用的方法来解决每个任务也是同样可行的。我们的基于测试的验证允许直接比较不同的解决策略。
可持续更新：虽然不是我们基准独有的，但我们的流水线是可重复的，因此基准可以持续更新。人工验证是次要的，每个任务验证时间不到10分钟。
4. 评估
我们在 GameDevBench 上评估了各种模型和智能体框架。
模型选择：从 Claude 系列模型中，我们评估了 Claude Haiku 4.5、Claude Sonnet 4.5 和 Claude Opus 4.5。从 Gemini 系列中，我们评估了 Gemini 3 Flash 和 Gemini 3 Pro。我们评估了 ChatGPT 系列的 ChatGPT Codex 5.1。对于开放权重模型，我们评估了 Qwen3-Vl-235B-Instruct 和 Kimi K2.5。
智能体框架选择：为了允许智能体访问项目文件和 Godot 应用程序本身，我们专注于本地运行的智能体框架。我们选择了命令行界面智能体框架，因为它们能够直接读取代码、图像和其他资产文件。我们在其各自的智能体框架中评估每个模型——Claude 模型用 claude-code，Gemini 模型用 gemini-cli，ChatGPT 模型用 codex。我们使用 OpenHands 评估 Kimi K2.5 和 Qwen3-Vl。我们还使用 OpenHands 评估 Claude Sonnet 4.5、Gemini 3 Flash 和 ChatGPT Codex 5.1，以比较不同框架间的性能。
4.1 多模态反馈
在这里，我们概述了两种工具配置，它们允许智能体通过编辑器截图和/或渲染视频从 Godot 访问更丰富的多模态信息。
基线：作为基线，每个智能体从项目目录内开始，并获得任务指令以及如何运行 Godot 的基本指令。我们提供额外方法支持智能体，主要是为了观察额外的视觉上下文是否提高性能。我们在附录 D 中提供了完整的提示。
编辑器截图 MCP：我们开发了一个 MCP 服务器，它加载当前任务的 Godot 编辑器，截取编辑器截图，然后将图像返回给智能体。这允许智能体查看游戏场景、节点树、节点检查器以及编辑器中存在的其他信息。此方法允许智能体利用额外的视觉反馈来验证其解决方案。
运行时视频：我们为智能体提供有关如何使用 Godot 内置录制功能生成游戏视频的指令，否则这些功能经常被忽略或误用。这与 MCP 服务器不同，因为它捕获 a) 仅在视频中存在的时间元素和 b) 当前相机视图（编辑器不显示相机视图）。通常，模型使用 python 将视频处理为图像帧，而不是直接摄取视频。
4.2 结果讨论
我们现在讨论在 GameDevBench 上评估智能体的发现（表2）。
游戏开发对即使是最有能力的模型也具有挑战性，并且当远离前沿时性能迅速下降。来自三个不同商业模型系列（GPT、Claude 和 Gemini）的最大模型在其原生智能体框架中，在没有额外多模态反馈的情况下，基线性能分别为34.1%、39.4%和46.2%。当我们远离前沿时，性能显著下降。Claude Haiku 4.5 解决了22.0%，Kimi K2.5 解决了34.1%，而 Qwen3-Vl-235B-Instruct 证明几乎完全无能力，仅解决了8.3%的任务。相比之下，Qwen3-Vl-235B-Instruct 在前端基准 Design2Code 中解决了92%的任务。
智能体性能在技能和编辑器类别上差异显著。我们观察到智能体在多模态要求更高的任务上表现更差的一般趋势（图5）。对于技能，智能体在游戏玩法任务上表现最好（平均成功率46.9%），在2D图形和动画上表现最差（平均成功率31.62%），这需要智能体理解图像或其他资产以进行动画和效果。编辑器类别之间的性能则取决于模型能力。整体表现最好的四个模型——Gemini 3 Pro、Gemini 3 Flash、Claude Opus 4.5 和 Claude Sonnet 4.5——在任务上表现相似，无论所需的编辑器类型如何。然而，Claude Haiku 4.5、ChatGPT Codex 5.1 和 Kimi K2.5 在需要场景和上下文编辑器的任务上表现更差，这通常比脚本任务具有更高的多模态要求。
多模态工具持续提高智能体性能。我们发现为智能体提供 MCP 或视频指令可以提高性能。这一趋势几乎适用于所有模型（Claude Haiku 4.5 例外，其性能保持相似）。然而，表现最佳的方法因模型而异。例如，Gemini 3 Flash 受益于 MCP，从47.0%提高到50.0%，而使用视频时为48.5%。另一方面，Claude Sonnet 4.5 使用视频从33.3%提高到41.7%，而使用 MCP 时没有改善。同时使用 MCP 和视频提供的额外好处可以忽略不计，其性能与任一方法的最佳性能相似。尽管所有任务都可以通过代码验证，但视觉反馈允许智能体验证和修正错误。这种行为强烈类似于最近的工作中看到的行为，其中视觉反馈提高了智能体性能。
智能体框架选择可能对性能产生重大影响，但效果因模型而异。我们使用原始框架和 OpenHands 评估了 Claude Sonnet 4.5、Gemini 3 Flash 和 GPT 5.1 Codex（表2）。我们观察到，当从其原生框架切换到 OpenHands 时，Claude Sonnet 4.5 和 GPT 5.1 Codex 的性能分别从33.3%提高到43.2%和从34.1%提高到45.5%。另一方面，Gemini 3 Flash 的性能从47.0%下降到36.4%，使其在其原生智能体框架中成为三个模型中最好的，而在 OpenHands 中最差。这可能是由于 Gemini 模型和 OpenHands 之间编辑工具不兼容。
成本因模型、框架以及是否提供多模态反馈而显著不同。我们发现启用多模态反馈几乎总是会增加成本以换取性能提高（图6）。唯一的例外是 Qwen3-Vl，它在该基准上挣扎。令人惊讶的是，同时使用这两种方法在所有 claude-code 智能体中更具成本效益，同时仍保持性能提升。我们怀疑模型能够在执行期间动态选择更有效的反馈机制。我们发现 Gemini 3 Flash 是最具成本效益的模型，而 Claude 模型往往是最不具成本效益的。我们观察到模型容量和每令牌成本并不一定与最终任务成本相关。例如，使用 claude-code 时，Claude Opus 4.5（图6，黄色）的成本是 Claude Sonnet 4.5（图6，绿色）的一半，尽管提供了更好的性能。再举一个例子，使用 OpenHands 时，Claude Haiku 4.5 的成本与 ChatGPT Codex 5.1 大致相同，尽管其每令牌更便宜且表现明显更差。
4.3 错误分析与改进方向
我们手动分析了智能体在解决任务时犯的一些最常见错误。这些错误表明能力上存在潜在差距，以及智能体开发的未来方向。虽然错误多种多样，但我们观察到两个一致的错误模式。
智能体在多模态理解上挣扎。也许最一致的错误模式源于多模态理解的缺乏。具体来说，要正确完成游戏开发任务，通常需要理解多模态输入。例如，创建（甚至只是选择）一个动画需要智能体解析多个图像或在精灵表中挑选特定的精灵。目前，智能体经常选择错误的图像或精灵（例如，选择行走动作精灵而不是攻击动作精灵）。显然，改进多模态理解将显著提高智能体游戏开发的性能。
智能体在常见游戏开发模式上挣扎。在游戏开发中，有许多常见的开发模式。例如，游戏元素（在 Godot 中称为节点）形成树结构，其中特定节点如 AnimatedSprite2D 和 CapsuleCollider 分别处理动画和物理。另一个例子是信号在满足条件时（例如，当两个碰撞器相交时）在各种文件之间触发。智能体经常将节点添加到树中的错误级别，丢弃必要的信号，或将资源分配给错误的元素。我们在附录 G 中提供了此类错误的示例。这强化了模型训练中长期存在的趋势——模型必须在特定领域上训练才能在该领域内表现出色。
5. 相关工作
智能体基准：软件开发一直是智能体开发的首要前沿之一。SWE-Bench 或许是第一个基准，也是迈向智能体软件开发的催化剂。随着时间的推移，已经开发了多个新的软件基准，但它们在很大程度上是单模态的。少数多模态软件基准主要关注前端 JavaScript 开发。相反，多模态智能体最常见的用例一直是计算机使用和网页导航。该领域的进展具有挑战性，因为智能体必须在动作空间中操作，而不仅仅是编写代码。游戏开发通过需要多模态输入但允许代码输出，弥合了这些领域之间的差距。GameDevBench 能够有效地从软件和计算机使用领域获得好处，从而实现有效的多模态评估。
游戏玩法：人工智能（AI）在游戏中的应用一直备受关注；游戏玩法被视为 AI 系统能力或智能的代理，项目范围从 Deep Blue、Alpha Go 和 Cicero 到更近期的通才如 SIMA 2。实际上，游戏提供了一个具有清晰奖励信号的交互式模拟环境，允许研究人员实验方法——特别是强化学习——以提高模型能力。最近大量使用 LLM 玩 Pokémon 的研究使用游戏玩家智能体来评估和探索前沿模型的智能体推理能力，然后直接用于游戏开发中以测试游戏。这种从作为游戏中的 NPC 和对手的游戏玩家智能体，转变为成为游戏开发过程一部分的转变，标志着对 GameDevBench 等基准的及时需求。
游戏开发：Concordia 及其后续关于桌面角色扮演游戏的工作试图用完全由与 LLM 的交互创建的高度适应性的故事来取代可交互的角色。其他工作试图完全取代游戏的物理引擎，以根据玩家动作立即生成帧。程序化内容生成在游戏资产创建和进化关卡设计中使用 AI 有着悠久的历史。然而，这些工作很大程度上关注游戏开发的单一侧面。最终，这些特性仍然需要在游戏引擎中组合以开发完整的游戏，而这正是 GameDevBench 直接评估的能力。
6. 结论
我们提出了 GameDevBench，这是第一个评估智能体解决游戏开发任务能力的基准。我们发现智能体在游戏开发任务上挣扎，特别是当任务需要更深层次的多模态理解时。前沿和非前沿模型之间的差距很大，绝对差异高达46.2%的 pass@1。此外，我们观察到智能体框架可以显著影响模型性能，尽管这高度依赖于模型和框架。最后，我们表明即使是提供多模态反馈的简单工具也能持续提高智能体性能，提供高达42%的相对改进。我们的发现强调了提高智能体多模态能力的必要性——无论是通过训练还是视觉反馈方法。我们推测，解决这些需求将提高智能体在软件和游戏开发甚至更广泛领域中的性能。
核心观点与主要内容总结
核心观点
游戏开发是评估复杂多模态智能体的理想领域：它天然结合了大规模软件开发的复杂性与对图像、动画、布局等静态及动态视觉元素的深度理解需求，且解决方案可通过代码确定性验证，避免了其他多模态任务评估中的主观性。
当前先进AI模型在游戏开发任务上能力不足：即使是最强的前沿模型（如 Gemini 3 Pro），整体任务通过率也仅略高于50%。性能随着模型能力下降而急剧衰退，且在多模态要求高的任务（如2D/3D图形）上表现远不如纯逻辑任务。
多模态理解是关键瓶颈：错误分析表明，智能体常在理解、选择和操作图像、动画帧等多模态资产上犯错，这是导致其在图形相关任务上失败的主要原因。
简单的多模态反馈机制可显著提升性能：为智能体提供编辑器截图或游戏运行视频作为视觉反馈，能使其自我纠正，从而普遍提高任务成功率，证明了提升模型多模态感知能力的有效性。
主要内容
提出了 GameDevBench 基准：这是首个专注于游戏开发领域的智能体能力评估基准。它包含 132个源自真实教程的任务，覆盖游戏玩法逻辑、3D/2D图形动画、用户界面四大类别，具有任务复杂、多模态、文件类型多样、解决方案可确定性验证等特点。
详细的基准构建流程：论文阐述了从视频和文本教程中，通过“数据准备 -> 自动任务构建 -> 自动精炼 -> 人工标注”四阶段流水线构建高质量任务集的完整方法，确保了任务的真实性、多样性和可解性。
全面的模型评估：论文评估了多个主流商业模型（Claude, Gemini, ChatGPT系列）和开源模型在两种智能体框架（原生框架和OpenHands）下的表现。评估不仅衡量总体成功率，还分析了不同技能类别、编辑器类型下的性能差异，以及成本效益。
有效的改进方法：提出了两种即插即用的多模态反馈工具：
编辑器截图MCP：让智能体能“看到”当前Godot编辑器状态。
运行时视频：让智能体能“看到”游戏实际运行效果。
实验证明这些方法能稳定提升各类模型的性能，最高提升幅度达42%。
深入的错误分析：指出了智能体失败的两大根本原因：多模态理解能力薄弱和缺乏游戏开发领域的特定知识模式，为未来研究指明了方向。
