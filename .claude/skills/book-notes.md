# 读书笔记生成

生成标准化的读书笔记，适用于 `readbooks` 项目。

## 参数

- `bookName`: 书名（必需）
- `author`: 作者（可选，会自动搜索）
- `detailLevel`: 详细程度（可选，默认为 `medium`，可选值：`simple`、`medium`、`detailed`）

## 示例

```
读书笔记 重构
读书笔记 "设计模式"
读书笔记 "代码整洁之道" detailed
```

## 模板路径

- 模板文件：`/mnt/d/readBooks/readbooks/templates/读书笔记模板.md`
- 笔记根目录：`/mnt/d/readBooks/readbooks/books/`
- **新书必须放到对应分类子目录下**，分类规则：
  - `software-engineering/` — 软件工程、编程语言、项目管理
  - `game-dev/` — 游戏开发、游戏引擎、图形渲染
  - `cs-fundamentals/` — 计算机基础、操作系统、编译原理、工具链
  - `thinking/` — 思维方法、认知科学、职业成长、非技术类

## 输出格式

生成的读书笔记包含以下部分：

1. **📚 基础信息** - 书名、作者、出版社、出版年份、标签等
2. **📖 内容概要** - 书籍简介、核心主题、主要章节
3. **🧠 知识架构** - Mermaid 思维导图
4. **✍️ 读书笔记** - 分章节的详细笔记，包含代码示例
5. **💡 个人思考** - 阅读过程中的思考和感悟
6. **🎯 实践应用** - 行动计划和实践建议
7. **💭 深度衍生思考** - 核心观点延伸、多角度分析、创新思考
8. **🔗 知识关联网络** - 与已读书籍的关联、概念映射、知识依赖关系
9. **📚 后续阅读路径规划** - 直接延伸、交叉验证、实践补充、个性化路径
10. **🎓 专家视角分析** - 多专家深度分析和观点整合
11. **📊 学习总结** - 收获、观念改变、未来行动
12. **📈 阅读进度** - 章节进度追踪

## 技术书籍笔记写作规范（强制遵守）

> 此规范针对技术类书籍（编程、游戏开发、软件工程、算法等），由实际反馈确立。

### 核心原则：概念和原理优先，代码为辅

1. **每章必须独立展开**：不能将多个章节合并。每章单独成节，深度展开其核心知识点，不能用一两句话带过。

2. **先讲"为什么"，再讲"怎么做"**：每个知识点先说明"是什么、为什么这样设计、解决了什么问题"，而不是直接给代码。

3. **架构图和设计图为主要表达方式**：
   - 优先使用 Mermaid 流程图、时序图、架构图说明系统设计
   - 优先使用对比表格说明不同方案的取舍
   - 优先使用 ASCII 框图展示层次结构和数据流
   - 能用图说清楚的，不用大段文字或代码

4. **代码仅作为辅助理解的工具**：
   - 代码出现前必须先有概念说明
   - 代码篇幅不超过章节总内容的 30%
   - 只有当概念本身必须用代码才能完整表达时才写代码
   - 禁止用代码替代概念解释（"看代码就知道了"是错误做法）

5. **对比分析是理解的最佳方式**：每个重要概念都应与相关领域的其他方案对比，揭示设计取舍背后的原因。

### 章节展开模板（技术书籍标准）

每章笔记应包含：
```
- 本章解决什么问题（背景和动机）
- 核心概念是什么（文字定义 + 架构图/流程图）
- 为什么这样设计（设计取舍分析 + 与其他方案对比表格）
- 各子概念之间的关系（Mermaid 关系图）
- 关键的工作原理（时序图或流程图，不是代码）
- 代码示例（仅辅助，简短说明关键接口即可）
```

## 执行步骤

1. **信息收集** - 使用 WebSearch 搜索书籍相关信息
2. **模板选择** - 根据书籍类型选择合适的模板
3. **专家分析** - 自动选择相关专家并生成深度分析
4. **知识关联** - 分析与已读书籍的关联关系
5. **后续推荐** - 基于书籍内容推荐后续阅读路径
6. **创建目录** - 创建书籍目录（如不存在）
7. **生成笔记** - 生成符合模板和上述技术书籍规范的完整读书笔记
8. **质量检查** - 验证：每章是否独立展开、架构图是否充分、代码是否只作辅助

- 生成个性化阅读路径
