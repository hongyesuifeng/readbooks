# 读书笔记生成

生成标准化的读书笔记，适用于 `readbooks` 项目。

## 参数

- `bookName`: 书名（必需）
- `author`: 作者（可选，会自动搜索）
- `detailLevel`: 详细程度（可选，默认为 `medium`，可选值：`simple`、`medium`、`detailed`）

## 示例

```
读书笔记 重构
读书笔记 "设计模式"
读书笔记 "代码整洁之道" detailed
```

## 模板路径

- 模板文件：`/mnt/d/readBooks/readbooks/templates/读书笔记模板.md`
- 笔记根目录：`/mnt/d/readBooks/readbooks/books/`
- **新书必须放到对应分类子目录下**，分类规则：
  - `software-engineering/` — 软件工程、编程语言、项目管理
  - `game-dev/` — 游戏开发、游戏引擎、图形渲染
  - `cs-fundamentals/` — 计算机基础、操作系统、编译原理、工具链
  - `thinking/` — 思维方法、认知科学、职业成长、非技术类

## 输出格式（v3.0 两大部分强制结构）

每篇读书笔记必须包含以下两大部分，不可跳过任何章节：

```
# Part 1：书籍内容梳理归纳
  ├── 📖 基本信息
  ├── 📝 内容概要（书籍定位、核心主题）
  ├── 🧠 知识架构（Mermaid mindmap）
  ├── ✍️ 分章笔记（每章独立展开，不可合并、不可跳过）
  │    每章格式：核心问题 → 关键概念 → 原理/机制 → 设计取舍 → 与其他章节的关联
  └── 章节速查表（可选）

# Part 2：深度分析与思考
  ├── D1 世界观提取（作者把什么当作理所当然？职业背景如何塑造了视野？）
  ├── D2 概念发生学（核心概念从哪来？经历什么演进？当前争议？）
  ├── D3 书籍间对话（指定1-2本对话对象，写出真正同意/分歧点及根因）
  ├── D4 反例检验（有什么成功案例违反了这个原则？说明了什么？）
  ├── D5 跨领域映射（游戏/软件/产品/管理中的对应物，准确度+启示）
  ├── D6 实践转化（具体到文件/类/函数/参数，及验证方式）
  ├── D7 元认知记录（读前认知→读后认知→关键转折点→残留疑问）
  └── 📊 学习总结（第5层认知改变 + 至少3个第4层洞见）
```

### Part 1 强制要求：
- **每章独立一节**——禁止合并多章为一段
- **先讲"为什么这样设计"，再讲"怎么实现"**
- **技术书：代码篇幅≤章节内容30%，架构图/流程图/对比表格优先**
- **非技术书：每章提炼核心论点+论证逻辑链，而非简单摘要**

### Part 2 强制要求：
- **D1-D7 七个维度必须全部覆盖**（理论书D6可弱化为"思维工具的应用场景"）
- **至少3个第四层分析（产生洞察）+ 1个第五层洞察（改变认知）**
- **禁止写法**："这本书让我收获很多""作者的观点很有启发性""可以借鉴到项目中"——必须具体到"收获了什么""启发了什么导致什么不同的行动""借鉴到哪个文件的哪个函数"
- 详见 `.claude/分析框架-深度阅读方法论.md`

## 技术书籍笔记写作规范（强制遵守）

> 此规范针对技术类书籍（编程、游戏开发、软件工程、算法等），由实际反馈确立。

### 核心原则：概念和原理优先，代码为辅

1. **每章必须独立展开**：不能将多个章节合并。每章单独成节，深度展开其核心知识点，不能用一两句话带过。

2. **先讲"为什么"，再讲"怎么做"**：每个知识点先说明"是什么、为什么这样设计、解决了什么问题"，而不是直接给代码。

3. **架构图和设计图为主要表达方式**：
   - 优先使用 Mermaid 流程图、时序图、架构图说明系统设计
   - 优先使用对比表格说明不同方案的取舍
   - 优先使用 ASCII 框图展示层次结构和数据流
   - 能用图说清楚的，不用大段文字或代码

4. **代码仅作为辅助理解的工具**：
   - 代码出现前必须先有概念说明
   - 代码篇幅不超过章节总内容的 30%
   - 只有当概念本身必须用代码才能完整表达时才写代码
   - 禁止用代码替代概念解释（"看代码就知道了"是错误做法）

5. **对比分析是理解的最佳方式**：每个重要概念都应与相关领域的其他方案对比，揭示设计取舍背后的原因。

### 章节展开模板（技术书籍标准）

每章笔记应包含：
```
- 本章解决什么问题（背景和动机）
- 核心概念是什么（文字定义 + 架构图/流程图）
- 为什么这样设计（设计取舍分析 + 与其他方案对比表格）
- 各子概念之间的关系（Mermaid 关系图）
- 关键的工作原理（时序图或流程图，不是代码）
- 代码示例（仅辅助，简短说明关键接口即可）
```

## 执行步骤

1. **信息收集** - 使用 WebSearch 搜索书籍相关信息
2. **模板选择** - 根据书籍类型选择合适的模板
3. **专家分析** - 自动选择相关专家并生成深度分析
4. **知识关联** - 分析与已读书籍的关联关系
5. **后续推荐** - 基于书籍内容推荐后续阅读路径
6. **创建目录** - 创建书籍目录（如不存在）
7. **生成笔记** - 生成符合模板和上述技术书籍规范的完整读书笔记
8. **质量检查** - 验证：每章是否独立展开、架构图是否充分、代码是否只作辅助

## 增强功能

### 自动专家分析
- 根据书籍类型自动选择相关专家
- 生成多专家视角深度分析
- 整合专家观点并生成综合结论

### 自动知识关联
- 分析书籍主题和关键词
- 查找已读书籍中的关联内容
- 生成知识关联网络和概念映射图

### 自动后续推荐
- 基于当前书籍主题推荐相关书籍
- 提供交叉验证和实践补充资源
- 生成个性化阅读路径

## 完成后的强制自我审查（必须执行）

笔记保存前，用以下问题检查，无法回答则修改：
1. 书里有没有哪个核心观点我觉得"有问题"或"有适用边界"？有没有写出来？
2. "未来行动"里有没有下周就能开始做的具体事？
3. 知识关联里，有没有两本书的观点存在**张力或矛盾**？
4. 哪个观点是真正新的认知（而不是"我早就知道了"）？有没有单独标出？
5. 整篇读完，有没有一个"让我对某件事的理解从X变成Y"的洞察？

如有问题，修改笔记再保存。详见 `.claude/rules.md`。

## 深度分析要求（v2.0 新增）

笔记初稿完成后，必须追加一份**深度分析补充**，覆盖以下七个维度（无法覆盖的说明原因）：

1. **D1: 世界观提取** — 作者把什么当作理所当然？职业背景如何塑造了他的盲区？
2. **D2: 概念发生学** — 每个核心概念的来源、演进、当前争议
3. **D3: 书籍间对话** — 指定1-2本已分析的书，写清楚在哪里同意、在哪里吵架
4. **D4: 反例检验** — 至少1个违反本书原则却成功的游戏案例
5. **D5: 跨领域映射** — 核心概念在软件/产品/管理中的对应物
6. **D6: 实践转化** — 具体到文件/类/函数/参数的行动计划
7. **D7: 元认知记录** — 读前认知→读后认知→转变触发点

详细框架见 `.claude/分析框架-深度阅读方法论.md`。

深度目标：每篇笔记至少 **3个第四层分析（产生洞察）+ 1个第五层洞察（改变认知）**。
