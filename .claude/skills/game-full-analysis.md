# 游戏全维度分析

<!-- 变更日志
2026-07-20: 初始创建 — 策划+程序双视角分析，支持源代码深度分析
-->

为游戏生成策划视角 + 程序视角的全维度深度分析文档。当提供源代码时，综合网上信息与源码内容，从游戏设计和工程实现两个角度进行系统性分析。

## 定位

本 skill 是 `game-notes` 的超集：
- **有源码时**：策划分析 + 技术分析 + 打通分析，三板块完整输出
- **无源码时**：自动降级为深度加强版策划分析（等价于增强版 `game-notes`）

## 参数

| 参数 | 必需 | 说明 |
|------|------|------|
| `gameName` | ✅ | 游戏名称 |
| `sourcePath` | 否 | 源代码根目录的绝对路径。提供后启用 Part 2（技术分析）和 Part 3（打通分析） |
| `platform` | 否 | 目标平台提示，辅助技术分析（如 PC/Android/iOS/Web） |

## 示例

```
/game-full-analysis 背包乱斗
/game-full-analysis 杀戮尖塔2 sourcePath=D:/projects/StS2
/game-full-analysis "黑神话悟空" sourcePath=E:/code/bmw platform=PC
```

## 模板路径

- 输出模板：`D:/readBooks/readbooks/templates/游戏全维度分析模板.md`
- 设计部分参考：`D:/readBooks/readbooks/templates/游戏分析模板.md`
- 质量规范：`D:/readBooks/readbooks/.claude/rules.md`
- 输出根目录：`D:/readBooks/readbooks/games/`

## 输出目录结构

```
games/{分类子目录}/{游戏名}/
  └── {游戏名}.md    # 全维度分析主文档
```

分类子目录规则（同 game-notes）：
- `action/` — 动作、格斗、射击、平台跳跃
- `rpg/` — 角色扮演、JRPG、ARPG、Roguelike
- `strategy/` — 策略、城市建设、塔防、RTS
- `puzzle/` — 解谜、益智
- `indie/` — 独立游戏、创意玩法（不好分类时放这里）

## 输出文档结构

```
# 《游戏名》全维度分析

📋 Part 0：基础信息与概览
  ├── 🎮 游戏基础信息
  │     ├── 开发/发行/平台/类型/时长
  │     └── 商业表现（销量/评价/收入，如能获取）
  ├── 💻 源码项目概览（有 sourcePath 时）
  │     ├── 技术栈总览表（引擎/语言/渲染/物理/网络/UI/构建，标注版本）
  │     ├── 目录结构（关键路径 + 每级目录职责一句话说明）
  │     └── 代码规模统计（文件数、行数估算、核心模块占比）
  └── 📊 整体评价总览（策划评分 + 技术评分 + 一句话总结）

🎨 Part 1：策划视角 — 游戏设计分析
  ├── 🎯 核心体验（一句话定位、核心循环图（单局+元循环）、3-5个记忆点）
  ├── 🧠 系统架构（Mermaid 架构图 → 每个主要系统按"设计目标/核心机制/深度来源/设计亮点"拆解）
  ├── 🎨 体验层分析（手感与操控、关卡/内容设计、叙事与世界观、美术与音乐）
  ├── ⚖️ 设计取舍分析（表格：决策 | 得到 | 放弃 | 约束条件）
  ├── 💡 值得借鉴的设计（每条落实到具体项目/系统/文件）
  └── ❌ 不足与问题（客观批判，含改进方向）

⚙️ Part 2：程序视角 — 技术架构与实现分析（有 sourcePath 时）
  ├── 技术栈总览与选型分析（每项技术：为什么选它、替代方案、选型背后的约束）
  ├── 项目架构
  │     ├── 模块划分 Mermaid 图（模块职责 + 依赖关系 + 数据流）
  │     ├── 核心设计模式识别（在代码中找到的具体模式和用途）
  │     └── 架构原则总结（分层/解耦/扩展性，附代码证据）
  ├── 核心系统实现分析（选 2-3 个最核心的深入源码级别）
  │     └── 每个系统：设计目标 → 代码结构 → 关键类/函数 → 数据流 → 状态管理 → 设计亮点 → 可优化点
  ├── 性能与工程实践
  │     ├── 性能优化手段（具体手段+代码证据）
  │     ├── 工程规范（命名/注释/测试/CI/文档）
  │     └── 可借鉴的工程实践
  ├── 技术亮点（独特的/巧妙的技术实现）
  └── 技术债与改进建议（客观评估 + 优先级排序）

🔗 Part 3：策划 ↔ 程序打通（有 sourcePath 时）
  ├── 设计意图 → 代码实现映射表（表格：策划需求 | 代码实现位置 | 实现方式 | 匹配度评估）
  ├── 技术约束 → 设计影响（技术限制如何反向塑造了设计决策）
  └── 设计文档 vs 代码现实（如有公开设计文档，对比实际实现的差异）

📊 Part 4：综合总结
  ├── 策划收获（作为设计师学到了什么）
  ├── 技术收获（作为工程师学到了什么）
  ├── 核心结论（2-3 句话总结本质）
  └── 🔗 知识关联
        ├── 与已读书籍的关联（每本书：具体的关联点，验证/挑战/延伸了什么观点）
        ├── 与其他游戏的关联（同类对比/设计传承/反例）
        └── 对自身项目的启发（具体到项目/文件/功能）
```

## 写作规范

### 策划部分（Part 1）继承 game-notes 全部规范

1. **系统优先**：每个系统先说明"设计目标是什么、解决了什么玩家体验问题"
2. **架构图为主要表达方式**：用 Mermaid 图 + 对比表格，避免流水账描述
3. **设计取舍必须有约束条件**：每个取舍要说清楚"被什么逼出来的"
4. **关联书籍知识**：将设计现象与已读书籍概念挂钩
5. **具体可行的借鉴**：每条借鉴落实到具体项目/文件/功能

### 技术部分（Part 2）专项规范

1. **选型必须说理由**：每项技术标注"为什么选它、替代方案是什么、约束条件是什么"
2. **架构用图说话**：用 Mermaid 图表达模块划分和依赖关系，附文字解释
3. **代码证据**：关键结论必须有代码位置证据（文件路径 + 简要说明），不能空口说
4. **架构原则追踪**：识别代码中实际遵循的架构原则，给出正例和违反的反例
5. **性能用数据**：不说"优化得好"，要说具体优化手段和推测的效果
6. **批判性立场**：不为源码辩护，客观指出过度设计、技术债、工程问题

### 打通分析（Part 3）专项规范

1. 每条映射必须有**匹配度评估**（完美匹配 / 基本匹配 / 有偏差 / 完全不同）
2. 偏差处要解释**为什么出现偏差**（技术限制？时间不够？设计变更？）
3. 至少找出 1 个"技术约束反向塑造设计"的案例（如果有）

### 通用深度要求

- 策划分析目标：≥3 个 L4（产生洞察），1 个 L5（改变认知）—— 层级定义见 rules.md
- 技术分析目标：≥2 个 L4
- 打通分析目标：≥1 个 L4
- 整体文档合计：≥3 个 L4 + 1 个 L5

## 禁止写法

### 策划部分禁止（同 rules.md 游戏分析禁止项）

- "X 系统设计得很好" —— 必须说为什么好、对谁好、在什么条件下好
- "值得借鉴的是 X 的设计思路" —— 必须说具体怎么借鉴、借鉴到哪个项目的哪个系统
- "与《书名》的 X 模式高度相关" —— 必须说具体异同
- 取舍只有优点没有真正代价 —— 每个决策都有代价

### 技术部分禁止

- "项目用了 XXX 框架" —— 必须说为什么选它、替代方案是什么
- "代码结构很清晰" —— 必须说用什么原则组织的、清晰在什么地方
- "性能优化做得好" —— 必须说具体手段、推测的效果
- "架构很优秀" —— 必须说什么场景下优秀、什么场景下可能有压力
- 堆砌文件名和类名 —— 代码证据应精选，用来说明观点而非填充篇幅

### 打通部分禁止

- "X 设计对应 Y 代码实现了" —— 必须有匹配度评估
- 只列举对应关系，没有偏差分析 —— 偏差往往比匹配更有价值

## 执行步骤

### Step 1：信息收集

同 `game-notes` 的多平台搜索策略：
- 必须按平台拆分搜索：Steam、TapTap、B站、App Store、九游、好游快爆、豌豆荚、微信小游戏、抖音小游戏
- 海外游戏加英文名搜索：Nintendo、Google Play、Metacritic 等
- 单次 WebSearch 返回空 ≠ 搜索工具不可用，换关键词/平台继续搜
- 对可访问 URL 用 WebFetch 提取页面信息
- 记录信息来源（每个平台搜到了什么/没搜到什么）

### Step 2：源码分析（有 sourcePath 时）

**2a. 目录结构总览**
```
- Glob 扫描 {sourcePath}/** 获取整体结构
- 识别关键目录（Assets/Scripts, src/, Source/ 等）
- 识别配置文件（*.sln, *.csproj, package.json, CMakeLists.txt, *.gradle 等）
- 输出：目录树 + 每级职责一句话
```

**2b. 技术栈识别**
```
- 读取配置文件识别技术栈
- 搜索关键 import/using/include 语句统计依赖
- 输出：技术栈总览表
```

**2c. 核心模块识别**
```
- 按目录结构识别模块边界
- Grep 搜索关键模式：
  - 状态机：StateMachine, FSM, State<, enum.*State
  - 事件系统：EventBus, MessageBus, Signal, Observer, delegate, event
  - 网络层：Network, Socket, RPC, Packet, Message
  - 资源管理：AssetBundle, ResourceManager, Addressable, Load
  - UI 系统：UIBase, UIView, Panel, Window, Screen
  - 数据层：SaveData, Config, Table, ScriptableObject
- 输出：模块列表 + 核心类/文件路径
```

**2d. 核心系统代码阅读**
```
- 选择 2-3 个最核心的系统深入阅读
- 理解类关系、数据流、状态管理
- 提取关键代码片段作为证据
```

### Step 3：生成 Part 1（策划分析）

按模板和规范生成策划视角分析，参考 `game-notes` 模板但做到更深：
- 设计取舍必须有约束条件列
- 记忆点必须具体，不只是列举
- 借鉴点必须落到具体项目

### Step 4：生成 Part 2（技术分析，有 sourcePath 时）

按模板和规范生成技术视角分析：
- 每个技术选型必须解释理由
- 架构图必须反映真实代码结构
- 核心系统分析必须包含代码证据
- 客观评估技术债

### Step 5：生成 Part 3（打通分析，有 sourcePath 时）

- 建立设计意图 → 代码实现的映射表
- 分析技术约束对设计的影响
- 识别设计文档和代码实现的偏差

### Step 6：生成 Part 4（综合总结 + 知识关联）

- 策划收获 + 技术收获分开写
- 知识关联必须具体，不能泛泛而谈
- 更新 `games/游戏知识图谱.md`

### Step 7：质量检查

**策划部分审查（rules.md 第一节 游戏分析自我审查问题）：**
- [ ] 最反直觉的设计决策是否点出来了？
- [ ] 一句话能向同行推荐"从这款游戏里学什么"吗？
- [ ] 有没有"看起来对但其实有问题"的设计被指出了？
- [ ] 每条借鉴能否对应到具体项目/文件/功能？
- [ ] 每个关联能否说出验证/挑战/延伸了什么观点？
- [ ] 取舍表格每行都有约束条件吗？
- [ ] 有没有我认为"取舍做错了"的决策？

**技术部分审查：**
- [ ] 技术选型每项都有理由和替代方案吗？
- [ ] 有没有更好的替代方案被遗漏了？
- [ ] 架构的核心原则一致贯彻了吗？有没有反例？
- [ ] 有没有过度设计被指出了？
- [ ] 技术债排序了吗？
- [ ] 工程实践有具体的可借鉴或应避免的建议吗？

**打通部分审查：**
- [ ] 每条映射有匹配度评估吗？
- [ ] 偏差处有解释原因吗？
- [ ] 至少找到 1 个"技术约束反向塑造设计"的案例吗？

**深度审查：**
- [ ] 策划 ≥3 L4 + 1 L5？
- [ ] 技术 ≥2 L4？
- [ ] 打通 ≥1 L4？

**禁止写法审查：**
- [ ] 逐条检查三部分的禁止写法清单
- [ ] 任何禁止写法出现 → 重写该段

### Step 8：保存 + 更新

- 将文档保存到 `games/{分类}/{游戏名}/{游戏名}.md`
- 更新 `games/游戏知识图谱.md`，添加新节点和关联
- 输出文件路径给用户

## 无源码降级行为

不提供 `sourcePath` 时：
- Part 0 的"源码项目概览"替换为简短说明"未提供源码"
- 跳过 Part 2（技术分析）、Part 3（打通分析）
- 输出结构：Part 0（基础）→ Part 1（策划）→ Part 4（总结+关联）
- 策划分析深度不降低，仍是 L3+ 级别
- Part 4 的"技术收获"替换为"技术观察"（基于网络信息做推测，标注为推测）

## 与 game-notes 的关系

`game-full-analysis` 是 `game-notes` 的超集：
- 无源码时输出等价于深度增强版 `game-notes`
- 有源码时额外输出 Part 2（技术）和 Part 3（打通）
- 不修改 `game-notes` skill，两者独立存在，按需选用
