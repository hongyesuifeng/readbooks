# 游戏分析笔记生成

生成标准化的游戏分析笔记，适用于 `readbooks` 项目的 `games/` 模块。

## 参数

- `gameName`: 游戏名（必需）
- `developer`: 开发商（可选，会自动搜索）

## 示例

```
游戏笔记 黑神话悟空
游戏笔记 "Hades"
游戏笔记 "空洞骑士"
```

## 模板路径

- 模板文件：`D:/readBooks/readbooks/templates/游戏分析模板.md`
- 笔记根目录：`D:/readBooks/readbooks/games/`
- **新游戏笔记必须放到对应分类子目录下**，分类规则：
  - `action/` — 动作、格斗、射击、平台跳跃
  - `rpg/` — 角色扮演、JRPG、ARPG、Roguelike
  - `strategy/` — 策略、城市建设、塔防、RTS
  - `puzzle/` — 解谜、益智
  - `indie/` — 不好分类的独立游戏、创意玩法游戏

## 输出格式

生成的游戏分析包含以下部分：

1. **🎮 基础信息** - 游戏名、开发商、平台、类型、游玩时长等
2. **🎯 核心体验** - 一句话定位、核心循环、记忆点
3. **🧠 系统架构** - Mermaid 架构图 + 主要系统逐一拆解
4. **🎨 体验层分析** - 手感、关卡设计、叙事、美术音乐
5. **⚖️ 设计取舍分析** - 表格形式呈现关键设计决策
6. **💡 值得借鉴的设计** - 具体可复用的设计点
7. **❌ 不足与问题** - 客观指出设计问题
8. **🔗 知识关联** - 与书籍的跨领域关联、与其他游戏的横向对比
9. **📊 总结** - 收获和核心结论

## 写作规范

### 核心原则：开发者视角为主

1. **系统优先**：每个游戏系统先说明"设计目标是什么、解决了什么玩家体验问题"，再描述机制细节。

2. **架构图为主要表达方式**：
   - 用 Mermaid 图展示系统间的依赖关系和信息流
   - 用对比表格说明设计取舍
   - 避免只是流水账式的游戏内容描述

3. **设计取舍分析必须有**：好的设计往往是在若干约束下的最优解，每个重要系统都要分析"得到了什么、放弃了什么"。

4. **关联书籍知识**：游戏分析的价值之一是将书本理论在真实产品中找到对应，尽量将设计现象与已读书籍中的概念挂钩。

5. **具体可行的借鉴**：借鉴点要写到能直接用于自己项目的程度，而不是泛泛而谈。

## 执行步骤

1. **信息收集** - 使用 WebSearch 搜索游戏信息、评测、设计分析文章
2. **分类判断** - 确定放入哪个子目录
3. **创建目录** - 在对应子目录下创建游戏目录
4. **生成分析** - 按模板生成完整游戏分析笔记
5. **知识关联** - 与已读书籍建立关联，更新 `games/游戏知识图谱.md`
