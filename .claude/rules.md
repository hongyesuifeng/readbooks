# 读书笔记与游戏分析质量规范

> 本文件定义笔记生成的批判性标准，每次生成笔记后必须用这些标准自我审查。
> 核心目标：每篇笔记都应该让未来的我读完后**改变了某个认知**，而不只是"记录了内容"。

---

## 一、生成笔记时的强制自我批判环节

每篇笔记完成初稿后，必须回答以下问题。如果无法回答，说明笔记不够深——回去修改，而不是原样保存。

### 游戏分析的自我审查问题

**关于核心洞察：**
- 这款游戏最反直觉的设计决策是什么？我有没有把它点出来，并解释为什么反直觉？
- 如果让我用一句话向游戏开发者同行推荐"从这款游戏里学什么"，我能说出来吗？（如果说不出来，总结章节不够深）
- 这款游戏有没有哪个设计是"看起来对但其实有问题"的？我有没有指出来？

**关于借鉴的真实性：**
- "值得借鉴"的每一条，我现在能说出对应到自己项目的**具体文件/系统/功能**吗？如果只是"可以参考这个思路"，说明借鉴没有落地。
- 有没有借鉴点是我写上去但自己其实并不确定真的有用的？那条应该删掉或标注"待验证"。

**关于知识关联的深度：**
- 每个书籍关联，我能说出"这款游戏验证/挑战/延伸了书里的哪个具体观点"吗？不能只是"都用了观察者模式"——要说这个游戏的用法和书里讲的有什么异同。
- 有没有这款游戏的某个设计，是和书里某个观点**相矛盾**的？矛盾本身往往比印证更有价值。

**关于设计取舍的深度：**
- 取舍表格里的每一行，我能说出"这个取舍是被什么约束条件逼出来的"吗？（预算？受众？技术？市场？）没有约束条件的取舍分析是空洞的。
- 有没有我认为"取舍做错了"的决策？如果全是"合理取舍"，说明分析不够批判。

---

### 读书笔记的自我审查问题

**关于真正的理解：**
- 书里有没有哪个核心观点，我读完后觉得"这个说法有问题"或"这在某些情况下不成立"？批判性不同意比全盘接受更有价值。
- 如果作者和另一本我读过的书的作者辩论，他们会在哪里分歧？我有没有构建这个张力？
- 书里哪个观点是我在读之前就隐约知道的？哪个是真正新的认知？两者要区分清楚。

**关于实践的真实性：**
- "未来行动"列表里，有没有下周就能开始做的具体事？如果全是"未来某天会……"，说明行动计划太虚。
- 书里哪个概念我现在能用一段话教给别人听懂？如果没有一个，说明理解还停留在表面。

---

## 二、笔记深度的层级标准

**第一层（不合格）：** 复述了内容——"这款游戏有X系统，它的工作方式是..."
**第二层（及格）：** 解释了设计——"X系统的设计目标是...，它解决了...问题"
**第三层（良好）：** 分析了取舍——"X系统得到了...，代价是...，这个取舍是被...约束逼出来的"
**第四层（优秀）：** 产生了洞察——"X系统揭示了一个普遍规律：当...时，...是比...更好的选择，因为..."
**第五层（卓越）：** 改变了认知——"读完这篇笔记，我对...的理解从...变成了...，这会影响我在...做出不同的决策"

**目标：每篇笔记至少有3个第四层分析，1个第五层洞察。**

## 三、command 和 skill 的迭代规范

### 何时更新 command/skill

每次生成完一篇笔记，必须回答：
- 这次生成过程中，有没有哪个步骤是多余的或顺序不对的？
- 有没有遇到信息缺口，是 command 没有要求搜索但应该搜索的？
- 生成的笔记有没有结构性缺陷，是模板本身的问题？

**如果以上有任何一个"有"，必须当场更新对应的 command 或 skill，不能只在脑子里记。**

### 迭代时的记录格式

在 command/skill 文件顶部维护一个变更日志，格式：
```
<!-- 变更日志
2026-06-17: 添加多平台搜索要求（教训：只查Steam漏掉国产手游）
2026-XX-XX: [下次改动的原因]
-->
```

## 四、跨笔记的知识积累规范

### 每完成3篇游戏分析后必须做的事

回顾已有的游戏分析，检查：
1. 有没有某个设计模式/原则在多款游戏里重复出现？如果有，把它提炼进 `games/游戏知识图谱.md` 的"设计模式追踪"
2. 有没有笔记里的"值得借鉴"在后来看来其实是错的？及时修正
3. 有没有两款游戏的对比，产生了新的洞察？写成跨游戏分析条目

### 每完成3本书后必须做的事

回顾已有的读书笔记，检查：
1. 有没有多本书在讲同一个底层原理，只是用了不同术语？把它们合并成一个核心概念条目放入知识图谱
2. 有没有两本书的核心观点存在张力或矛盾？构建这个对话
3. `books/知识关联图谱.md` 是否反映了最新的阅读积累？

## 五、禁止出现的写法

这些写法出现即降级，必须重写：

**游戏分析禁止：**
- "这款游戏的X系统设计得很好" — 没有说为什么好，对谁好，在什么条件下好
- "值得借鉴的是X的设计思路" — 没有说具体怎么借鉴，借鉴到哪里
- "与《书名》的X模式高度相关" — 没有说这款游戏的用法和书里讲的有什么具体异同
- 设计取舍表格里只有优点没有真正的代价 — 每个决策都有代价，找不出来说明分析不够深

**读书笔记禁止：**
- "这本书让我收获很多" — 说出具体收获了什么
- "作者的观点很有启发性" — 启发了什么，改变了什么认知
- 每章笔记长度均等 — 重要的章节应该比不重要的章节长得多
- "未来会在项目中应用" — 说出具体在哪个项目、哪个功能、怎么应用

## 六、深度分析七维度框架（强制使用）

> 详见 `.claude/分析框架-深度阅读方法论.md`

### 笔记结构强制要求（v3.0）

每篇读书笔记必须分为两大部分，缺一不可：

```
# Part 1：书籍内容梳理归纳 — 客观呈现"书上说了什么"
# Part 2：深度分析与思考 — "这意味着什么，对不对，怎么用"
```

Part 1 要求：**每章独立一节，不可合并，不可跳过。** 技术书代码≤30%，非技术书每章提炼核心论点+论证链。

Part 2 要求：**D1-D7 七个维度全部覆盖，不可跳维度。**

### 七维度检查表

| 维度 | 核心问题 | 输出要求 |
|------|---------|---------|
| **D1: 世界观提取** | 作者把什么当作理所当然？ | 一句话世界观 + 逻辑链 + 反面何时成立 |
| **D2: 概念发生学** | 核心概念从哪来？经历了什么演进？ | 概念名 → 来源 → 演进节点 → 当前争议 |
| **D3: 书籍间对话** | 这本书与另一本在哪里同意、在哪里吵架？ | 指定对话对象 + 共识点 + 分歧点及根因 |
| **D4: 反例检验** | 有什么成功游戏违反了这个原则？ | 反例游戏名 + 违反了什么 + 为什么能成功 |
| **D5: 跨领域映射** | 这个概念在软件/产品/管理中叫什么？ | 映射目标领域 + 准确度 + 相互启示 |
| **D6: 实践转化** | 下个Sprint能改哪一行？ | 项目/文件/类/函数 + 具体改动 + 验证方式 |
| **D7: 元认知记录** | 读完我什么变了？ | 读前认知 → 读后认知 → 关键转折点 |

所有笔记分析深度必须达到至少 **3个第4层（产生洞察）+ 1个第5层（改变认知）**。

## 七、适用范围

本规范适用于：
- `/game-notes` command 执行时
- `/book-notes` skill 执行时
- 任何手动整理游戏或书籍分析笔记时

**每次笔记生成完成后，必须用本文件的第一节"强制自我批判环节"逐项检查，不通过则修改笔记再保存。**
