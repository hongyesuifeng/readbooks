## 📝 内容概要

### 书籍定位

《游戏编程设计模式》不是GoF的搬运工——它从游戏开发的**真实痛点**出发，重新推导出一套"在游戏中有效的设计模式"。Nystrom 在 EA 做了多年引擎开发，核心洞察：**游戏和其他软件的根本区别不是"性能要求更高"，而是"设计在整个开发周期中会持续剧烈变动"**。Web加功能=加API，游戏改设计="主角现在可以飞了"——后者的代码改动是维度的变化。

全书20个模式按"你的游戏代码在哪疼"分为五类。每个模式结构：意图→动机→模式→何时用→注意事项→代码示例→设计决策→相关模式。

### 核心主题

1. **设计模式重访** — 命令/享元/观察者/原型/单例/状态：GoF模式在游戏中的正确用法
2. **序列模式** — 双缓冲/游戏循环/更新方法：游戏的"心跳"怎么组织
3. **行为模式** — 字节码/子类沙箱/类型对象：100种怪物不写100个类
4. **解耦模式** — 组件/事件队列/服务定位器：不让音频/AI/UI搅在一起
5. **优化模式** — 数据局部性/脏标记/对象池/空间分区：帧率掉到30fps的救命稻草

## ✍️ 分章笔记

### 第1章：架构、性能和游戏

**Nystrom 的全书宣言**："架构是用来让你的代码更容易改的——因为游戏设计中唯一确定的是，一切都会变。"

他不像 Uncle Bob 从"依赖倒置"讲架构。Nystrom 的架构标准只有一个：**当策划下周说"让主角能飞"，你改几行代码？**

**"好架构"的四个度量**：改动范围（改多少文件）、改动难度（每个改动的复杂度）、引入 bug 的概率、团队沟通成本。

**"性能 vs 抽象"的张力**：游戏不能为了"干净"牺牲性能。但 Nystrom 的立场是先写干净，然后 Profiler 告诉你哪里需要去掉抽象。大多数"性能优化"是过早优化。

---

### 第2章：命令模式（Command）

**定义**："命令 = 物化的方法调用（A reified method call）。"把"做一件事"变成"一个对象"。

**三大应用**：

**① 按键配置** — 不是 `if (key == SPACE) player.jump()`，而是每个按钮持有 `Command*`，按下时调用 `execute()`。`JumpCommand` 和 `FireCommand` 是独立的类。玩家随意改键——输入层完全不知道"跳跃"是什么，只知道"有个命令，执行它"。

**② 玩家与AI指令统一** — AI 产生 `MoveCommand`，手柄也产生 `MoveCommand`——同一个命令类，不同生产者。Demo/回放/AI 操控玩家角色零额外代码。

**③ 撤销（Undo/Redo）** — `MoveUnitCommand` 存下 `xBefore_`/`yBefore_`，`undo()` 恢复之前的位置。`command_history` 是一个数组，Undo = pop 最近一个并调 `undo()`。**整个时间回溯系统 = 一个数组 + 一个指针。**

**设计决策**：命令可以是单例（无状态的 `JumpCommand`——所有角色共享同一个实例）或每次 new（`MoveUnitCommand`——存储特定单位的特定时刻）。

---

### 第3章：享元模式（Flyweight）

**核心思想**：把对象数据拆成**内在状态**（各实例共享的，上下文无关的）和**外在状态**（每个实例独有的）。

**经典案例：渲染一片森林**。10000棵树，如果一个 `Tree` 对象包含完整的 Mesh + Texture，内存爆炸。用享元：一个 `TreeModel`（内在：共享的 Mesh、纹理、shader）被所有10000棵树共享；每棵树实例只存位置、颜色、缩放（外在：独有的）。

**Nystrom 强调**：享元在 GPU 硬件层面有直接支持——**实例化渲染（Instanced Rendering）**就是用享元思想设计的图形API。

**代价**：外在状态的参数一旦确定就不能再对共享的内在状态产生影响（所有树用同一个 Mesh——你没法让其中一棵树"比别的树多一片叶子"）。如果外在状态需要频繁修改内在，享元不适用。

---

### 第4章：观察者模式（Observer）

**动机案例**：成就系统。"从桥上摔下去"的成就由物理引擎触发，但物理引擎不应该知道"成就系统"的存在。

**核心实现**：
```cpp
// 物理引擎只管发通知——不知道谁在听
notify(player, EVENT_START_FALL);
// 成就系统注册成为观察者
physics.addObserver(achievementSystem);
```

**Nystrom 的辩护**：

批评① "太慢了"——发通知就是遍历链表 + 调虚函数。在非热点路径上根本不算什么。

批评② "太多动态分配"——Nystrom 用**侵入式链表**解决：每个 `Observer` 自身包含 `next_` 指针，自己就是链表节点。零额外分配。代价：一个观察者同一时刻只能观察一个主题。

**"垂死呼吸"技巧**：Subject 析构前给所有 Observer 发"我马上要死了"的通知——Observer 立刻注销自己，避免悬空指针。

**"失效监听者"（Lapsed Listener）问题**：GC 语言中，忘记注销的观察者永远不会被回收，且每次通知浪费 CPU。**纪律比技巧重要。**

**与事件队列的区别**：观察者 = 同步（通知→当场处理）。事件队列 = 异步（把消息扔进队列，接收者什么时候处理是自己的事）。

---

### 第5章：原型模式（Prototype）

**核心思想**：不要 new 一个对象——**克隆一个已有对象**。

**游戏中的场景**：怪物生成器。不同种类的怪物有不同的初始属性（生命、攻击、速度）。如果用工厂模式，每种怪物需要一个工厂类。用原型：一个 `Spawner` 持有一个 `Monster* prototype`——调用 `prototype->clone()` 即可生成新怪物，不需要知道具体是什么怪物。

**Nystrom 的扩展讨论**：JavaScript 等原型继承语言（没有 class，对象直接克隆对象）如何在游戏中建模数据。克隆一个"模板哥布林"→改它的属性→这就是新类型——不需要 class 声明。

**何时不用**：当对象的构造逻辑很复杂（需要初始化各种子系统）时，克隆可能产生"不完整的对象"——原型不知道被克隆之后还需要做什么。

---

### 第6章：单例模式（Singleton）

**Nystrom 花在这章"警告"上的篇幅比"怎么用"还多。**

单例 = 一个类只有一个实例，全局可访问。常见的游戏单例：`AudioManager::instance()`、`Game::instance()`。

**为什么游戏开发喜欢单例**：游戏有大量"全游戏只有一个"的东西（音频系统、渲染器、输入管理器）。用单例比传参数方便——你不用把 `Audio*` 从 main 一路传到每个需要播放音效的类。

**Nystrom 的警告**：
- 单例 = 全局变量 + 语法糖。全局变量让代码难以理解、难以测试、线程不安全
- "全游戏只有一个"不等于"永远只有一个"——今天一个音频系统，明天可能加语音聊天、背景音乐系统
- **替代方案**：传参、从基类获取、放在已有的单例 `Game` 里、用服务定位器（至少可以替换实现）

**何时真的需要**：当你确信"这个类在整个程序生命周期中绝不会需要第二个实例，且全局访问确实大幅简化了代码"时——这个条件比你想的更难满足。

---

### 第7章：状态模式（State）

**核心思想**：用一个对象代表一个状态，替代 `switch(heroine.state)` 地狱。

**经典案例**：平台游戏女主角。状态：站立、跳跃、蹲下、俯冲。

```cpp
class HeroineState { virtual void handleInput(Heroine&) = 0; };
class StandingState : public HeroineState { ... };
class JumpingState  : public HeroineState { ... };
// heroine.state_->handleInput(heroine);  // 一行替代整个switch
```

**Nystrom 展现状态机的四种进化**：

| 变体 | 能力 | 适用场景 |
|------|------|---------|
| **基本FSM** | 每个状态一个类 | 简单角色行为 |
| **并发状态机** | 两个独立状态机同时跑 | "拿着枪时能跳吗"——一个状态机管"在做什么"，一个管"拿着什么" |
| **层次状态机** | 状态可以继承父状态 | "蹲下时开枪"继承"站立时开枪"的行为 |
| **下推状态机** | 用栈记录状态历史 | 从"瞄准"退出时自动回到"站立"而非"跳跃" |

**局限**：当状态数爆炸（交互式的状态组合）时，FSM 的类数量也爆炸。这时需要考虑行为树或 GOAP。

---

### 第8章：双缓冲（Double Buffer）

**核心思想**：在任何时刻，消费者看到的永远是一个完整的、一致的状态。写入在"后缓冲"进行，写完后原子交换→后缓冲变成新的前缓冲。

**三个经典应用**：
1. **图形渲染**：前缓冲 = 屏幕正在显示的帧，后缓冲 = 正在绘制的帧。交换 = VSync
2. **游戏状态快照**：AI读取"当前状态"做决策，物理写入"下一帧状态"。帧末原子交换——保证AI基于一致状态
3. **任何需要"原子性更新多个值"的场景**：如 UI 布局计算

**代价**：内存翻倍（需要两套缓冲）。对于内存敏感的移动设备，需要只在必要的子系统使用。

---

### 第9章：游戏循环（Game Loop）

**核心思想**：游戏是实时驱动的——即使玩家什么都不做，帧也在推进。游戏循环 = `processInput()` → `update()` → `render()`。

**Nystrom 展开的三种实现**：

| 实现 | 描述 | 问题 |
|------|------|------|
| **简单无限制循环** | `while(true) { update(); render(); }` | 帧率不可控——快电脑跑太快，慢电脑跑太慢 |
| **固定帧率+Sleep** | 计算帧耗时，不够则 Sleep 补齐 | 如果 update 超过帧预算，游戏整体变慢 |
| **固定更新步长+可变渲染** ⭐推荐 | 游戏逻辑按固定 Δt（如 1/60s）更新；渲染用插值平滑运动 | 主要方案的轻微复杂度 |

**"追赶"机制**：如果一帧的 update 耗时太长（CPU 被其他进程抢占），下一帧的循环会"追赶"——连续调用多次 update 直到游戏时间追上墙上时间。但如果 update 本身就慢，追赶 = 恶性循环。

---

### 第10章：更新方法（Update Method）

**核心思想**：世界中的每个实体都有一个 `update(Δt)` 方法，每帧被游戏循环调用一次。

```cpp
class Entity { virtual void update(double dt) = 0; };
// 游戏循环：for (auto& e : entities_) e->update(dt);
```

**解决的问题**：游戏循环不需要知道"世界中有哪些种类的实体"——只需遍历一个 `vector<Entity*>` 并调用 `update()`。

**Nystrom 的实用警告**：
- 在遍历过程中不要增删实体——用"待删除列表"延迟到遍历结束后处理
- 可变时间步（dt 不稳定）→ 物理计算变得非确定性 → 固定 Δt 比可变更安全
- `update()` 应该放在实体本身、组件中、还是委托？取决于架构选择

---

### 第11章：字节码（Bytecode）

**全书最复杂的模式**。

**核心思想**：不给设计师暴露 C++ 代码——给他们一套小型的、安全的"技能编程语言"。你的引擎运行一个微型虚拟机来解释这些字节码指令。

**经典场景**：魔法技能系统。不写 `class Fireball : public Spell { ... }`（每加一个技能 = 改引擎代码 + 重新编译）。而是定义一套指令集：
```
INST_SET_HEALTH  // 设置血量
INST_ADD         // 加法
INST_LITERAL 25  // 字面量 25
```
"治疗25点" = `[INST_SET_HEALTH, INST_ADD, INST_LITERAL, 25]`——一个字节数组。

**Nystrom 展开的实现**：基于栈的虚拟机（指令推入/弹出操作数栈）→ 对比寄存器虚拟机 → 如何表示值（tagged union / 接口 / 变体）→ 代码生成的前端（手写字节码 vs 图形化编辑器 vs 小型脚本语言）。

**代价**：维护一个 VM 是额外的工作量。只有当你的游戏有足够多的技能/行为（50+），字节码才比手写C++值得。

---

### 第12章：子类沙箱（Subclass Sandbox）

**核心思想**：基类提供一整套"protected 安全操作"——派生类只能通过这些操作来实现行为。

**案例**：超级英雄游戏。`Superpower` 基类提供：
```cpp
protected:
    void move(Vector2 where);       // 安全移动
    void playSound(int soundId);    // 安全播放
    void spawnParticles(int type);  // 安全粒子
```
派生类 `FireballPower` 的 `activate()` 方法只能用这些 protected 方法——不能访问引擎内部、不能操作内存、不能崩掉游戏。**新人写一个新技能 = 只写 fill 一个 `activate()` 方法。**

**何时不用**：当基类提供的"安全操作"不够表达新需要的功能时——要么不断扩展基类（最终基类变巨无霸），要么打破沙箱规则（模式失效）。

---

### 第13章：类型对象（Type Object）

**核心思想**：让**类的实例**充当**类型的模板**——你不创建新类，你创建新的数据。

**经典案例**：怪物系统。100种怪物不需要100个 `class Orc, class Goblin, class Dragon...`。一个 `Monster` 类 + 一个 `Breed` 类：
```cpp
class Breed {
    int health_;
    const char* attack_;
    Breed* parent_;  // 继承：哥布林→兽人→怪物基类
};
Monster::Monster(Breed& breed) : health_(breed.getHealth()) { }
```
"兽人"是一个 `Breed` 实例（health=100, attack="slash"），不是 `class Orc`。**新增怪物类型 = 新增一行数据，不碰代码。**

**JSON/XML 友好**：类型对象天然适合从配置文件加载——设计师在 JSON 里调怪物属性，程序员不需要在场。

---

### 第14章：组件模式（Component）⭐ 全书最重要

**动机**：`Bjorn` 类（平台游戏角色）包含输入、物理、碰撞、动画、渲染、音效——5000行，没人敢动。

**解决方案**：把 `Bjorn` 沿领域边界切开：
```
Bjorn = InputComponent + PhysicsComponent + GraphicsComponent + SoundComponent
```
组件之间**互相不知道对方存在**。物理程序员和图形程序员各行其是。

**"为什么继承不够"的经典论证**：
```
装饰品 = 可见 + 无碰撞    区域 = 不可见 + 有碰撞    道具 = 可见 + 有碰撞
继承：GameObject→{Zone, Decoration}→Prop 该继承谁？菱形陷阱！
组件：一个 GameObject，加或不加 GraphicsComponent，加或不加 PhysicsComponent。
```
**组件 = 自助餐，继承 = 固定套餐。**

**组件间通信三种策略**（通常同时使用）：
1. 修改容器共享状态（最解耦，但处理顺序关键）
2. 组件间直接引用（简单直接，引入耦合）
3. 消息广播（灵活，性能开销大）

**极端解耦**：实体变成 `int id`，组件各自存在连续数组中。ECS的前身。

---

### 第15章：事件队列

**与观察者的本质区别**：观察者 = 同步（通知→当场处理）。事件队列 = 异步（消息扔进队列，什么时候处理是你的事）。

**三大应用**：
- **音频**：播放音效→扔进队列→音频线程消费→发者不等
- **分帧处理**：每帧从队列拿 N 个事件——突发巨量事件不拖死帧率
- **网络**：收到的包→队列→主循环消费

**"光环"警告**：事件队列让消息流时序不透明。你不知道你发送的事件会在哪一帧被处理。

---

### 第16章：服务定位器（Service Locator）

**核心思想**：一个 `Locator` 类持有一个全局可访问的服务引用——但具体是哪个实现，可以在运行时替换。

```cpp
class Locator {
    static Audio* audio_;  // 抽象接口，不是具体实现
public:
    static Audio* getAudio() { return audio_; }
    static void provide(Audio* service) { audio_ = service; }
};
```

**比单例强在哪**：可以在不修改任何调用代码的情况下，替换整个音频系统的实现。测试时注入一个 `NullAudio`（什么都不做），发布时注入 `RealAudio`。

**Nystrom 的警告**：服务定位器仍然是全局状态——只是比单例多了一层解耦。首选还是依赖注入（通过构造函数传入），服务定位器是"当你没法重构整个项目"的妥协方案。

---

### 第17章：数据局部性（Data Locality）

**Nystrom 的当头棒喝**：你以为瓶颈是算法 O(n²)，实际是 CPU 等内存（Cache Miss）。

**核心原理**：L1缓存 ≈ 1ns，主存 ≈ 100ns。CPU 不是按字节取数——是按**缓存行**（64字节）取。如果64字节里有16个连续排列的4字节整数，一次取数全进缓存。如果分散在16个不同对象中（因为指针），16次取，每次miss = 1600ns。

**Nystrom 的实验**：把游戏实体的组件从"堆上分散"改为"连续数组存储"——**同样算法，速度提升50倍**。

**实践**：`vector<PhysicsComponent>` 替代 `list<GameObject*>`，遍历时所有数据紧挨着，CPU预取完美命中。

---

### 第18章：脏标记（Dirty Flag）

**核心思想**：不要每帧都重新计算——**只在"结果被需要且输入已改变"时计算**。

**经典案例**：场景图的变换矩阵。每个节点有本地变换→世界变换。每帧重算所有节点的世界变换很昂贵。用脏标记：当节点的本地变换改变时，设置 `dirty_ = true`。渲染前递归检查——如果未脏，用缓存的世界变换；如果已脏，重算并清除标记。

**关键细节**：脏标记需要级联传播——一个节点脏了，它所有子节点的世界变换也间接脏了（因为世界变换 = 父世界 × 本地）。

**何时不用**：如果"计算"本身很快（比检查标记+分支预测失败的开销还快），脏标记反而变慢。

---

### 第19章：对象池（Object Pool）

**为什么需要**：`new Bullet()` → `fly()` → `delete`。高频创建/销毁的代价：malloc（去堆上找空闲内存，慢）、构造/析构（不一定慢但多余）、内存碎片化（越运行越慢）。

**解决方案**：预分配 `Bullet pool[1000]`。需要时从池取一个未使用的，`activate()` 重置状态。用完标记"未使用"，放回池。**子弹对象从不 new 也不 delete——永远在池中循环。**

**与数据局部性的协同**：如果池是连续数组，所有活跃子弹在内存中紧挨——Cache Miss降到最低。

---

### 第20章：空间分区（Spatial Partition）

**核心思想**：你不是每帧检查"每对物体是否碰撞"（O(n²)）。你把世界按位置分成格子——每个格子记录"谁在里面"。碰撞检测只检查"同一个格子或相邻格子中的物体"。

**最简实现**：固定网格。世界分 M×N 格，每格一个链表存该格的单位。战斗检测：遍历每个格子中的单位→只与本格+相邻8格中的单位比对。

**Nystrom 展开的多种结构**：
- 固定网格（最简单，适用于均匀分布的小单位）
- 四叉树/八叉树（适用于大小差异巨大的对象）
- BSP树（Doom/Quake 时代的经典——对2D/3D空间二分）
- k-d树（任意维度）

**关键细节**：物体穿越网格边界时，需要从旧格子移除→插入新格子。大物体可能跨多个格子。"攻击范围"可能大于所在格子——需要检查相邻格子。

### 20章速查

| 模式 | 一句话 | 不用时的后果 |
|------|--------|------------|
| 命令 | 把"做什么"变成对象 | 按键/回放/AI/撤销全耦合 |
| 享元 | 共享重复的不可变数据 | 10000棵树 = 10000份Mesh在内存 |
| 观察者 | 广播"发生了什么"不在乎谁听 | 物理引擎导入成就系统的头文件 |
| 原型 | 克隆已有对象代替new | 每怪物类型一个工厂类 |
| 单例 | 慎用——全局访问的最后手段 | 全局状态+耦合+不测试+不线程安全 |
| 状态 | 对象替代switch-case地狱 | 状态越多switch越大 |
| 双缓冲 | 原子切换 | 渲染撕裂/状态读取不一致 |
| 游戏循环 | 分离时间步长和渲染 | 快电脑快进/慢电脑慢放 |
| 更新方法 | 每帧调一次update | 游戏循环 = 巨大switch on type |
| 字节码 | 小型安全编程语言 | 每个技能改C++→重新编译 |
| 子类沙箱 | 基类提供保护的safe操作 | 新人写了段崩溃代码 |
| 类型对象 | 实例作类型模板 | 100怪物=100个类 |
| 组件⭐ | 实体=N个独立组件 | 5000行Bjorn地狱 |
| 事件队列 | 异步解耦+分帧 | 突发巨量事件拖垮帧率 |
| 服务定位器 | 可替换的全局访问 | 硬编码具体实现→无法测试 |
| 数据局部性 | 连续存储>指针跳转 | Cache Miss拖死性能 |
| 脏标记 | 变了再算，没变用缓存 | 每帧重复算几百个变换矩阵 |
| 对象池 | 复用不新建不销毁 | 碎片内存+GC抖动 |
| 空间分区 | 按位置排，只查附近 | O(n²)碰撞检测 |

# Part 2：深度分析与思考

## D1: 世界观提取 — Nystrom 把什么当作理所当然？

**Nystrom 的底层信念**：游戏代码腐烂的根源是**代码的组织方式跟不上设计变化**。他衡量架构的好坏只有一个标准——"当策划下周说让主角能飞时，你改多少代码？"

这源自他的 EA 引擎开发经历——代码被几十个策划使用，每周有人来问"角色能不能做X"——他必须说可以而不需要重写引擎。

**反面——什么时候 Nystrom 的模式是过度设计**：Game Jam、一次性发布、单人项目。组件模式的"允许多人并行开发"优势归零。

---

## D2: 概念发生学 — Nystrom 真正原创的是什么

Nystrom 被误解为"搬运GoF"。真正的贡献是**重新分类**：GoF 按"代码在做什么"分类（创建/结构/行为），Nystrom 按"游戏代码在哪疼"分类（序列/行为/解耦/优化）。这个分类揭示了一个GoF没有覆盖的问题空间——**在游戏特有的时间压力、类型爆炸和性能约束下，哪些结构是可维护的。**

"组件模式"是 Nystrom(2014) → Unity GameObject+Component → Godot Node树 → 现代ECS的演进轴心。

---

## D3: 书籍间对话

### Nystrom vs Fowler《重构》

| | Nystrom | Fowler |
|--|---------|--------|
| 代码烂了怎么办 | 一开始选对模式——预防 | 烂了之后系统性地修——治疗 |
| 实践中 | 前半开发用Nystrom选架构，后半用Fowler修积累的坏味道 ||

### Nystrom vs Godot

Godot 完美实现了 Nystrom 核心：`_process(delta)`=更新方法、`signal`=观察者的标准实现、场景=组件模式。但 Godot 的 `Autoload` = 服务定位器——Nystrom 警告过的全局状态陷阱。这是一个有意识的工程权衡。

---

## D4: 反例检验

**反例1: 《Celeste》** — 3600个手工房间，纯手工代码。设计从不变化→不需要引擎级架构。

**反例2: 《矮人要塞》** — 反模式代码 + 有史以来最复杂的模拟。Tarn 优化的是"这个计算值不值得做"——不是 Nystrom 的"同样计算用更少周期"。

|-------------|----------|
| 组件 | React Hooks——组合超越继承 |
| 命令 | Redux Actions——可回放的状态变更 |

| 对象池 | DB Connection Pool——复用昂贵资源 |
| 数据局部性 | CPU Cache——游戏对全软件行业的核心输出 |

## D6: 实践转化 — Godot 直接应用

### 命令模式 → 时间回溯
```gdscript
class MoveCommand:
    var target; var from_pos; var to_pos
    func execute(): target.global_position = to_pos
    func undo():    target.global_position = from_pos
var history: Array[MoveCommand] = []; func undo(): history.pop_back().undo()
```

### 观察者 → Godot Signal
```gdscript
# 反模式: player.gd 直接调 enemy.damage_player(10)
# Nystrom式: enemy.gd 声明 signal attack_landed(damage)
# player.gd: enemy.attack_landed.connect(_on_damage_received)
```

### 对象池 → 子弹
```gdscript
class BulletPool:
    var _pool: Array[Bullet] = []
    func get() -> Bullet:
        return _pool.pop_back().reset() if _pool.size() > 0 else bullet_scene.instantiate()
```

## D7: 元认知记录

**读前**："设计模式是Java的23个，在游戏里挑几个用。"

**读后**："Nystrom 重新定义了什么是游戏设计模式——从游戏真实痛点独立推导，而非GoF翻译。分类方式本身就是创新。衡量模式好坏的标准：不是优雅，是策划改需求时的改动成本。"

## 📊 学习总结

**第5层认知改变**：对"设计模式"的理解从"让代码更优雅的规则"变为**"在特定变化模式下最小化改动成本的工程决策"**。

**3个第4层洞见**：
1. Nystrom 的痛点分类取代GoF结构分类——这是模式思想在游戏领域的独立进化
2. 组件模式= "加新行为只需加新文件"——React Hooks 10年后同一思想的另一语法
3. Godot Signal 是观察者在游戏中的标准实现："广播者不知道谁在听"
